# Gate 2 房屋物件與揭露主線設計

## 目標

Gate 2 要把平台主體從「房源刊登」推進到「房屋物件」。房源只是市場上的出租/出售刊登資料；房屋物件才是權屬、現況說明、重大事項揭露、屋主保證與後續鏈上 hash 的承載主體。

本階段完成後，流程應該是：

`屋主身份啟用` -> `建立房屋物件` -> `填寫財產/現況說明書` -> `確認屋主重大事項保證` -> `產生揭露快照與 hash` -> `物件達可刊登狀態` -> `選擇上架出租或賣屋` -> `補完出租/出售刊登資料` -> `正式公開到對應列表`

`READY_FOR_LISTING` 只代表物件本體已完成足夠揭露，可以被拿來建立刊登；它不代表物件已公開，也不會自動出現在出租或賣屋列表。

## 平台責任邊界

平台是媒合與資訊揭露平台，不是政府機關、地政士、法院、公證人、鑑價機構或第三方官方認證單位。

平台負責盡力提供：

- KYC 實名制與角色身份基礎，提高平台可信度
- 文件上傳、預覽、浮水印與留存管道
- 資料完整度檢核
- 屋主聲明、重大事項揭露與附件摘要
- disclosure snapshot 與 hash，增加可追溯性
- 讓交易雙方更容易查核、比對與留證的流程

平台不負責保證：

- 使用者上傳文件必然真實
- 權狀、謄本、稅籍、證照或其他文件具備官方驗證效力
- 屋主聲明或屋況揭露必然完整無誤
- 交易雙方後續履約、隱匿、詐欺、不實揭露或違約行為

建議全站統一說明：

> 平台透過 KYC 實名制、角色認證、文件揭露、浮水印與 hash 留存，提高媒合流程的便利性、透明度與可信度；但文件內容、屋況聲明、交易承諾與後續履約責任，仍由提供資料與參與交易的使用者自行負責。

## 角色資料比對原則

屋主與仲介身份流程保留 OCR 與智能審核，但定位必須是「使用者填寫資料與附件內容的一致性檢查」，不是平台對證件、權狀、證照或法律資格做官方認證。

角色啟用判斷應使用下列語意：

- `資料比對通過`
- `佐證資料已提供`
- `角色資料完整度達標`
- `資料不足或比對不一致，請補件`

避免使用下列容易越界的語意：

- `官方認證通過`
- `文件真實性已驗證`
- `權狀有效`
- `證照有效`
- `平台保證`

屋主與仲介的最低通過標準：

- 必填欄位不可空白
- 重要欄位需能從使用者輸入或附件 OCR 中取得可比對線索
- 附件上傳後必須套用平台用途限制浮水印
- OCR 不一致時只能判定為資料不足或需補件，不判定文件真假

租客身份維持輕量流程：填寫工作/收入資訊，薪資證明或收入證明可選上傳；若有上傳，顯示已提供收入佐證。租客流程不走屋主/仲介的 OCR 比對主線，送出通過必要欄位檢核後即可啟用租客角色。

## 文件與預覽浮水印原則

凡是涉及證件、證照、權狀、稅籍、財產說明書、收入證明、薪資證明或其他可能具有法律效益的上傳檔案與預覽圖，都必須加上浮水印。

短版浮水印：

> 僅供本平台媒合使用，非官方驗證文件

完整版說明：

> 本文件由使用者自行提供，平台僅作為媒合資訊揭露與留存管道，不代表平台、政府機關或第三方專業單位已驗證文件真實性、完整性或法律效力。請交易雙方自行查證並視需要委託專業人士確認。

浮水印需求：

- 圖片預覽必須在可視區域覆蓋浮水印
- PDF 或不可直接處理的檔案至少要在預覽容器上覆蓋浮水印
- 下載或開新視窗前仍需顯示用途限制說明
- 後端可先保存原檔，但前端預覽不可裸露敏感文件

## 核心模型

### 房屋物件 `properties`

房屋物件是平台內的資產型主體，至少包含：

- 物件序號
- 所有人 `owner_user_id`
- 來源屋主身份申請 `source_credential_submission_id`
- 地址
- 權狀字號或產權文件編號
- 權屬 hash `deed_hash`
- 物件狀態
- 揭露完成度狀態
- 建立時間與更新時間

物件狀態：

- `DRAFT`：物件已建立，但揭露資料尚未完成
- `VERIFIED`：權屬來源與揭露流程已在平台流程中完成
- `REJECTED`：物件資料或聲明未通過平台流程

### 房源刊登 `listings`

房源刊登是市場上的公開或待公開資訊，必須綁定一筆 `properties`。同一筆物件未來可以有不同刊登紀錄，但本階段先限制同一時間只允許一筆主要草稿或公開刊登。

刊登至少包含：

- 刊登序號
- `property_id`
- 刊登目的 `list_type`：`UNSET / RENT / SALE`
- 刊登狀態 `status`：`DRAFT / ACTIVE / ARCHIVED`
- 刊登完整度 `setup_status`：`INCOMPLETE / READY`
- 公開標題
- 公開地址顯示層級
- 價格資訊
- 刊登條件
- 更新時間

`list_type = UNSET` 代表物件已建立，但使用者尚未選擇要出租或出售。此狀態不得公開。

#### 出租刊登 `RENT`

出租刊登需補完租賃條件，才可從草稿進入可公開狀態：

- 月租金
- 押金
- 管理費
- 最短租期
- 可入住日
- 出租坪數
- 房廳衛
- 樓層
- 車位
- 家具設備
- 是否可開伙
- 是否可養寵物
- 是否可申請租金補貼
- 是否可設籍
- 是否歡迎學生、青年、長者
- 看房方式與聯絡偏好
- 出租補充說明

#### 出售刊登 `SALE`

出售刊登需補完交易條件，才可從草稿進入可公開狀態：

- 開價
- 是否含車位
- 車位價格或車位補充說明
- 建坪
- 主建物坪數
- 附屬建物坪數
- 土地坪數
- 格局
- 樓層
- 屋齡
- 目前使用狀態
- 交屋條件
- 產權或交易補充說明
- 看屋方式與聯絡偏好

### 財產/現況說明書

正式欄位之後會依使用者提供的現有版本調整；本階段先建立結構與必要分類。

建議分類：

1. 基本建物資訊
   - 建物型態
   - 建物用途
   - 登記坪數
   - 使用坪數
   - 樓層與總樓層
   - 屋齡
   - 格局
   - 車位
   - 管理費

2. 權利與使用狀態
   - 是否有共有部分
   - 是否有抵押
   - 是否有查封、假扣押、法院拍賣或其他限制
   - 是否出租中
   - 是否遭第三人占用
   - 是否存在未揭露租約或使用權
   - 是否有管委會或社區規約限制

3. 建物異動
   - 是否增建
   - 是否改建
   - 是否違建
   - 是否有用途變更
   - 是否有未登記部分

### 屋主重大事項保證

重大事項不可空白送出。每一項建議使用 `YES / NO / UNKNOWN`，避免使用者不知道時硬填 `NO`。

本階段先納入：

- 是否海砂屋
- 是否輻射屋
- 是否凶宅
- 是否曾發生火災
- 是否漏水
- 是否壁癌
- 是否結構損壞
- 是否有違建
- 是否有產權爭議
- 是否欠稅、漏稅或稅籍異常
- 是否遭查封、假扣押或法院拍賣程序
- 是否有未揭露租約或占用
- 是否有管委會、社區規約或使用限制
- 其他應揭露事項

每個保證項目包含：

- 回答：`YES / NO / UNKNOWN`
- 補充說明
- 可選附件

`YES` 或 `UNKNOWN` 應要求補充說明。`NO` 可不填說明。

## 揭露快照與 hash

`disclosure_snapshot_json` 是物件揭露在某一版本的固定快照，不是任意備註。

快照至少包含：

- 物件基本資料
- 權屬資料摘要
- 財產/現況說明書
- 屋主重大事項保證
- 附件摘要
- 平台責任邊界文字版本
- 屋主確認時間
- 快照版本

`disclosure_hash` 由穩定 JSON 序列化後計算 SHA-256。

`deed_hash` 由 owner、地址、權狀字號、來源申請與必要權屬摘要計算 SHA-256。

## 完整度狀態

物件需有明確完整度狀態：

- `BASIC_CREATED`：已由屋主身份啟用建立基本物件
- `DISCLOSURE_REQUIRED`：等待財產/現況說明書
- `WARRANTY_REQUIRED`：等待重大事項保證
- `SNAPSHOT_READY`：揭露快照已產生
- `READY_FOR_LISTING`：物件可支援建立出租或出售刊登

房源正式公開條件：

- 房源綁定 `property_id`
- 物件狀態為 `VERIFIED`
- 物件完整度為 `READY_FOR_LISTING`
- disclosure snapshot 與 hash 已存在
- 房源 `list_type` 已選擇 `RENT` 或 `SALE`
- 該刊登目的的必要欄位完整
- 房源 `setup_status` 為 `READY`
- 使用者按下正式公開動作後，房源 `status` 才可變為 `ACTIVE`

## 後端流程

### 屋主身份啟用

屋主身份啟用通過後：

1. 從申請 payload 取得地址與權狀字號。
2. 建立 `properties` 基本資料。
3. 初始完整度為 `DISCLOSURE_REQUIRED`。
4. 建立或綁定一筆 listing 草稿。
5. listing 草稿不可上架，直到 property 完整度達 `READY_FOR_LISTING`。
6. property 達 `READY_FOR_LISTING` 後，listing 仍維持草稿，等待使用者選擇出租或出售並補完刊登資料。

### 物件揭露更新

新增 property API：

- `GET /api/properties/mine`
- `GET /api/properties/:id`
- `PUT /api/properties/:id/disclosure`
- `POST /api/properties/:id/disclosure/confirm`

`PUT /disclosure` 儲存財產/現況說明書與重大事項保證。  
`POST /confirm` 檢核必要欄位、產生 snapshot/hash，並將完整度推進到 `READY_FOR_LISTING`。

### 刊登分流與公開

新增或調整 listing API：

- `PUT /api/listings/:id/intent`
- `PUT /api/listings/:id/rent-details`
- `PUT /api/listings/:id/sale-details`
- `POST /api/listings/:id/publish`

`PUT /intent` 設定刊登目的為 `RENT` 或 `SALE`。  
`PUT /rent-details` 僅接受出租刊登欄位。  
`PUT /sale-details` 僅接受出售刊登欄位。  
`POST /publish` 需同時檢核 property ready 與 listing ready，通過後才將 `status` 改為 `ACTIVE`。

## 前端流程

### 我的房源 `/my/listings`

卡片除了房源狀態，也要顯示物件完整度：

- 物件已建立
- 待填財產說明
- 待確認重大事項
- 揭露已完成
- 可建立刊登
- 待選擇出租或出售
- 出租刊登待補資料
- 出售刊登待補資料
- 已公開

### 草稿詳情 `/my/listings/:id`

草稿頁應清楚分成兩層：

- 物件揭露狀態
- 刊登目的
- 出租或出售刊登資料

當物件未達 `READY_FOR_LISTING` 時，刊登目的與公開按鈕不可用，並提示先完成物件揭露。

當物件已達 `READY_FOR_LISTING` 時，頁面顯示兩個主要動作：

- `上架出租`
- `上架賣屋`

選擇後才顯示對應刊登表單。出租與出售不可共用同一組必要欄位。

### 物件揭露編輯

本階段可以先放在草稿詳情內，不一定獨立做 `/my/properties/:id`。

需要有：

- 財產/現況說明書區塊
- 屋主重大事項保證區塊
- 文件附件區塊
- 平台責任邊界確認
- 產生揭露快照的確認動作

### 公開房源

公開列表只顯示 `status = ACTIVE` 的 listings。物件達 `READY_FOR_LISTING` 但尚未完成出租/出售刊登時，不得出現在公開出租列表或賣屋列表。

公開上架後可顯示摘要：

- 物件已完成揭露
- 屋主已確認重大事項
- 是否有重大瑕疵揭露
- deed hash 短碼
- disclosure hash 短碼
- 平台不做官方背書說明

公開列表需依刊登目的分流：

- 出租列表只顯示 `list_type = RENT`
- 賣屋列表只顯示 `list_type = SALE`
- 若維持單一 `/listings` 入口，至少要有明確的出租/賣屋篩選與頁面標題

## 本階段不做

- 不接完整 `PropertyRegistry` 鏈上交易流程
- 不做仲介授權
- 不做 case / stake
- 不替文件內容做官方認證
- 不做完整物件 marketplace
- 不做信義房屋式房仲服務保障或品牌背書
- 不做自動估價或房貸試算

## 驗收條件

1. 屋主身份啟用後會建立 property 與 listing 草稿。
2. listing 草稿綁定 property。
3. property 有財產/現況說明書資料結構。
4. property 有重大事項保證資料結構。
5. 重大事項不可全空白確認。
6. `YES` 或 `UNKNOWN` 的重大事項需要補充說明。
7. 確認揭露後產生 deterministic `disclosure_hash`。
8. 未完成 property 揭露前，listing 不可上架。
9. 文件預覽有平台用途限制浮水印。
10. 前端明確顯示平台是媒介與揭露管道，不做官方背書。
11. property 達 `READY_FOR_LISTING` 後不會自動公開。
12. 使用者需選擇出租或出售後，才可補完對應刊登資料。
13. 出租刊登必要欄位未完成時不得公開到出租列表。
14. 出售刊登必要欄位未完成時不得公開到賣屋列表。
15. 公開列表不得顯示 `list_type = UNSET` 或 `status != ACTIVE` 的草稿。
16. 屋主與仲介 OCR 文案明確表示資料比對與佐證，不宣稱官方認證。
