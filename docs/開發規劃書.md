## 一、目前專案狀態

### 1.1 第一階段完成（2026-04-19）

第一階段正式主線已全部完成：

1. `/kyc` 自建 eKYC onboarding（含進度接續精確步驟復原）
2. `/login` 使用 `person_hash + wallet SIWE + password`
3. `/forgot-password` 忘記密碼 / 初次設定密碼
4. `/member` 身份中心顯示自然人 KYC 與 IdentityNFT 狀態
5. `/profile` 會員資料頁（Email / 手機 / 通訊地址 OTP 變更）
6. `/listings` 房源列表 + 詳情（KYC VERIFIED 才能刊登）
7. 預約看房系統（排隊、確認、屋主鎖定洽談）

### 1.2 已完成能力

#### Auth

- `POST /api/auth/login`
- `POST /api/auth/password/set`（onboarding 完成後初次設密碼）
- `POST /api/auth/reset-password/request-otp`（忘記密碼 / 密碼未設定 → 發 OTP 至 email）
- `POST /api/auth/reset-password/set-password`（OTP 驗證後設定新密碼）
- `GET /api/auth/me`
- `POST /api/auth/logout`
- `POST /api/auth/wallet/change`
- session-only cookie + server-side session
- SIWE nonce / verify
- 未完成設密碼即離頁 → 自動 logout（OnboardingPage unmount cleanup）

#### eKYC / Onboarding

- Email OTP
- Phone OTP
- 身分證正反面上傳
- Google Vision OCR（姓名、身分字號、生日、發證日期、發證地點、地址、父母姓名、性別推導）
- 自拍人臉比對（AWS Rekognition）
- 第二證件上傳
- 錢包綁定 + SIWE
- `person_hash` / `identity_hash`
- Go OPERATOR key 自動 mint `IdentityNFT`
- indexer 將 `users.kyc_status` 同步為 `VERIFIED`
- KYC 409 衝突提示（Email / 手機 / 身分重複）
- `EMAIL_NOT_ACTIVATED`：Email 已完成 KYC 但密碼未設 → 前端自動導向 `/forgot-password`
- 接續進度精確步驟復原（`calcResumeWizardStep` 依 session 路徑欄位計算真實步驟）
- 所有 OCR 欄位存入 `kyc_sessions`（含 gender / issue_date / issue_location / father_name / mother_name）

#### 會員資料 / User Profile

- `GET /api/user/profile`
- Email / 手機 / 通訊地址 OTP 修改（6 支 API）
- Admin KYC 人工審核入口（`/admin/kyc/pending`）

#### 房源系統（Listing）

- KYC VERIFIED 限制刊登
- 房源 CRUD（create / update / publish / remove / close）
- 狀態機：DRAFT → ACTIVE → NEGOTIATING → LOCKED → SIGNING → CLOSED
- 上架費用：NT$40 / 天
- 預約看房：排隊 / 確認 / 已看 / 有意願 / 取消
- 屋主手動鎖定洽談（NEGOTIATING）/ 解鎖

#### Database

- `users`、`kyc_sessions`、`kyc_submissions`
- `otp_codes`、`auth_nonce`、`wallet_session`、`user_credentials`
- `listings`、`listing_appointments`
- `task_blockchain_logs`（足跡）
- `indexer_checkpoints`、`processed_events`

## 三、開發治理準則

### 3.1 開發順序（必須遵守）

1. DB schema 先定義
2. 合約事件與函式介面定義
3. Go service / indexer / signer 對接
4. React `api` 與頁面整合

### 3.2 安全原則

- 嚴禁 hardcode 私鑰、RPC URL、第三方 API key
- 所有敏感資訊走 `.env`
- 所有鏈上寫入結果以 event / indexer / DB 為準
- 前端不應自己假設交易已成功
- 關聯式資料庫規格以 `docs/database/relational-database-spec.*` 為單一準據

### 3.3 架構原則

- `platform/*` 只提供基礎能力，不承擔業務決策
- `modules/*` 才是業務模組邏輯所在
- `db/repository/*` 只做資料存取，不塞商業規則
- 文件只能描述現況，未來藍圖需明確標示為 phase

- `platform/storage`
- `platform/ocr`
- `platform/faceai`
- `platform/notify`

### 4.2 Indexer 現況

目前 indexer 的正式用途只有：

- 同步 `IdentityNFT.IdentityMinted`
- 將 `users.kyc_status` 更新為 `VERIFIED`
- 回填 `identity_nft_token_id`

目前只有 `modules/user/event_handler.go` 為正式 worker。其餘 property / agency / listing / case 事件同步仍屬下一階段。

### 4.3 合約主線

目前身份主線以 `IdentityNFT.sol` 為準：

- tokenId `1`：NATURAL_PERSON
- tokenId `2`：OWNER
- tokenId `3`：TENANT
- tokenId `4`：AGENT

其中目前 live flow 已實際使用的是 `NATURAL_PERSON`。

## 六、下一階段開發方向

### 6.1 第二階段：身分認證完成（角色申請）

目標：讓已 KYC 的使用者能申請取得平台角色，解鎖對應功能。

#### 6.1.1 開發項目

1. **OWNER 申請流程**
   - `IdentityCenterPage` 加入申請入口
   - Go `modules/user` 新增 `ApplyOwnerRole`：寫入 `user_credentials`（status=PENDING）
   - Admin 審核通過後，OPERATOR key mint `IdentityNFT tokenId=2`
   - indexer 同步，`user_credentials.status = ACTIVE`
   - 前端身份中心顯示屋主憑證與其專屬功能

2. **TENANT 申請流程**
   - 類似 OWNER，tokenId=3，可能不需要文件審核（待確認）

3. **AGENT 申請流程**
   - tokenId=4，需要執照文件（後續 phase 再評估）

#### 6.1.2 DB 規格（已存在，待接線）

| Table | 用途 |
|-------|------|
| `user_credentials` | 角色狀態（PENDING / ACTIVE / REVOKED）、tokenId、mintTxHash |
