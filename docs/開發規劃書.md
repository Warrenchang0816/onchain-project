## 一、目前專案狀態

### 1.1 第一階段完成（2026-04-19）

第一階段正式主線已全部完成：

1. `/kyc` 自建 eKYC onboarding（含進度接續精確步驟復原）
2. `/login` 使用 `person_hash + wallet SIWE + password`
3. `/forgot-password` 忘記密碼 / 初次設定密碼
4. `/member` 身份中心顯示自然人 KYC 與 IdentityNFT 狀態
5. `/profile` 會員資料頁（Email / 手機 / 通訊地址 OTP 變更）
6. `/listings` 房源列表 + 詳情（KYC VERIFIED 才能刊登）
7. 預約看房系統（排隊、確認、屋主鎖定洽談）

### 1.2 已完成能力

#### Auth

- `POST /api/auth/login`
- `POST /api/auth/password/set`（onboarding 完成後初次設密碼）
- `POST /api/auth/reset-password/request-otp`（忘記密碼 / 密碼未設定 → 發 OTP 至 email）
- `POST /api/auth/reset-password/set-password`（OTP 驗證後設定新密碼）
- `GET /api/auth/me`
- `POST /api/auth/logout`
- `POST /api/auth/wallet/change`
- session-only cookie + server-side session
- SIWE nonce / verify
- 未完成設密碼即離頁 → 自動 logout（OnboardingPage unmount cleanup）

#### eKYC / Onboarding

- Email OTP
- Phone OTP
- 身分證正反面上傳
- Google Vision OCR（姓名、身分字號、生日、發證日期、發證地點、地址、父母姓名、性別推導）
- 自拍人臉比對（AWS Rekognition）
- 第二證件上傳
- 錢包綁定 + SIWE
- `person_hash` / `identity_hash`
- Go OPERATOR key 自動 mint `IdentityNFT`
- indexer 將 `users.kyc_status` 同步為 `VERIFIED`
- KYC 409 衝突提示（Email / 手機 / 身分重複）
- `EMAIL_NOT_ACTIVATED`：Email 已完成 KYC 但密碼未設 → 前端自動導向 `/forgot-password`
- 接續進度精確步驟復原（`calcResumeWizardStep` 依 session 路徑欄位計算真實步驟）
- 所有 OCR 欄位存入 `kyc_sessions`（含 gender / issue_date / issue_location / father_name / mother_name）

#### 會員資料 / User Profile

- `GET /api/user/profile`
- Email / 手機 / 通訊地址 OTP 修改（6 支 API）
- Admin KYC 人工審核入口（`/admin/kyc/pending`）

#### 房源系統（Listing）

- KYC VERIFIED 限制刊登
- 房源 CRUD（create / update / publish / remove / close）
- 狀態機：DRAFT → ACTIVE → NEGOTIATING → CLOSED / REMOVED / EXPIRED
- `LOCKED / SIGNING` 目前僅為 schema / enum 預留，正式鏈上 case 流程尚未接線
- 上架費用：NT$40 / 天
- 預約看房：排隊 / 確認 / 已看 / 有意願 / 取消
- 屋主手動鎖定洽談（NEGOTIATING）/ 解鎖

#### Database

- `users`、`kyc_sessions`、`kyc_submissions`
- `otp_codes`、`auth_nonce`、`wallet_session`、`user_credentials`
- `credential_submissions`
- `listings`、`listing_appointments`
- `task_blockchain_logs`（足跡）
- `indexer_checkpoints`、`processed_events`

## 二、當前正式方向

### 2.1 資料流基準

```text
┌──────────────────────────────────────────────────────────┐
│                   Ethereum / Sepolia                      │
│              （Source of Truth，鏈上狀態為準）             │
└────────────────────┬─────────────────┬────────────────────┘
                     │ emit events      │ send tx（MetaMask）
              ┌──────▼──────┐    ┌──────▼──────┐
              │ Go Indexer  │    │   前端       │
              │（訂閱 events）│    │（用戶 MetaMask）│
              └──────┬──────┘    └──────┬──────┘
                     │ 同步 DB          │ HTTPS / WSS
              ┌──────▼──────────────────▼──────┐
              │            Go-service           │
              │   REST API + WebSocket + Crypto │
              └──────────────┬─────────────────┘
                             │ SQL
              ┌──────────────▼─────────────────┐
              │          PostgreSQL             │
              │    （索引層、查詢快取、文件存放）  │
              └────────────────────────────────┘
```

- 鏈上合約與 events 是最終真相來源，DB 是 read model 與 off-chain workflow 中繼層。
- 目前 live repo 已正式接線的是 `IdentityNFT` + `IdentityMinted` indexer。
- `listings` / `listing_appointments` 仍主要由 DB workflow 驅動，`CaseTracker / ListingStakeVault` 為後續 phase。
- `WebSocket / WSS` 在架構上保留，但目前 repo 尚未正式接上 router / hub。

### 2.2 四層建構邏輯

| 層級 | 定義 | 本專案對應 |
|------|------|------------|
| 基礎設施層 | DB、RPC、雲服務、檔案儲存、環境變數 | `infra/`、`go-service/internal/platform/*` |
| 鏈上合約層 | Solidity、Events、Roles、SBT / Vault / Registry | `task-reward-vault/contracts/*` |
| 服務橋接層 | Go API、indexer、signer、session、KYC orchestration | `go-service/internal/modules/*` |
| 用戶互動層 | React SPA、wallet、KYC wizard、身份中心、房源 UI | `react-service/src/*` |

### 2.3 目前正式主線

第一階段已完成；第二階段的 Gate 1A 角色認證閉環已接線，下一步才是 Gate 1B 權限切換：

**第二階段主線：身分與角色閉環（Gate 1A 已交付）**

- `/credential/owner`、`/credential/tenant`、`/credential/agent` 已接上正式頁面與 API
- Go `modules/credential` 已提供送件、檔案上傳、智能審核、人工審核請求、啟用與 admin review API
- `credential_submissions` 為角色申請 write model，`user_credentials` 為已啟用角色 read model
- IdentityNFT tokenId=2（OWNER）/ 3（TENANT）/ 4（AGENT）於使用者點擊啟用時才 mint
- `CredentialMinted` indexer 已回寫 PostgreSQL，身份中心可重新整理後仍讀到角色狀態
- Gate 1B 權限切換仍未啟用，刊登與預約規則目前維持 Gate 0 的 `KYC VERIFIED`

### 2.4 目前不是主線的部分

以下內容目前不是正式主線，不應再被文件描述成 live flow：

- TWID callback KYC
- WebSocket notifier / hub
- `property / agency` 的正式 Go module 化
- 鏈上 CaseTracker / ListingStakeVault 接線

### 2.5 正式主幹 Gate Roadmap（2026-04-22 起適用）

自 2026-04-22 起，平台主體後續開發正式採用 gate-based roadmap 治理。上位治理文件為：

- `docs/superpowers/specs/2026-04-22-platform-mainline-gate-roadmap-design.md`

主幹 Gate 順序如下：

1. `Gate 0`：基線收斂
2. `Gate 1`：身分與角色閉環
3. `Gate 2`：Property On-chain
4. `Gate 3`：Agency On-chain
5. `Gate 4`：Case On-chain
6. `Gate 5`：Stake 正式上鏈
7. `Gate 6`：Production Readiness 前置

補充治理邊界：

- 正式主幹採 `租屋主線`；買賣路線只做每個 Gate 的架構預留與前置條件，不作為目前主幹正式需求
- 中程鏈上化順序固定為 `Property -> Agency -> Case -> Stake`
- `OWNER 自營` 與 `OWNER -> AGENT 授權` 兩條路線並存，不得讓仲介主線吃掉 owner-first 的平台定位
- `WebSocket / notify / TWID / AI 審核` 屬支援軌，可視 Gate 需要附帶交付，但不作為主幹 Gate 阻塞條件

## 三、開發治理準則

### 3.1 開發順序（必須遵守）

1. DB schema 先定義
2. 合約事件與函式介面定義
3. Go service / indexer / signer 對接
4. React `api` 與頁面整合

### 3.2 安全原則

- 嚴禁 hardcode 私鑰、RPC URL、第三方 API key
- 所有敏感資訊走 `.env`
- 所有鏈上寫入結果以 event / indexer / DB 為準
- 前端不應自己假設交易已成功
- 關聯式資料庫規格以 `docs/database/relational-database-spec.*` 為單一準據

### 3.3 架構原則

- `platform/*` 只提供基礎能力，不承擔業務決策
- `modules/*` 才是業務模組邏輯所在
- `db/repository/*` 只做資料存取，不塞商業規則
- 文件只能描述現況，未來藍圖需明確標示為 phase

### 3.4 團隊使用方式（GitHub Actions）

目前團隊開發流程應以 PR 為主：

1. 從 feature branch 開發並推送到 GitHub
2. 開 PR 到 `main`
3. 等待 `Go CI`、`Frontend CI`、`Contracts CI`、`Security` 完成
4. checks 全綠後再進行 code review / merge

治理建議：

- 後續應在 GitHub `Settings > Branches > Branch protection rules` 將 `Go CI`、`Frontend CI`、`Contracts CI`、`Security` 設為 `main` 的 required checks。
- required checks 屬於 GitHub repo 設定，不會存放在程式碼倉庫內；本文件只記錄建議治理方式與目前 workflow 名稱。

### 3.5 Gate-based 開工規則（2026-04-22 起）

自本次 roadmap 定稿後，所有主幹開發都必須先對照 `docs/superpowers/specs/2026-04-22-platform-mainline-gate-roadmap-design.md` 確認目前所在 Gate，再依序完成：

1. `Spec`
2. `Implementation Plan`
3. `Implementation`
4. `Verification`

補充規則：

- 未完成當前 Gate 驗收前，不得把下一個 Gate 寫成正式進行中的主線
- 每個 Gate 的正式驗收至少要同時具備 `Spec evidence`、`Implementation evidence`、`Verification evidence`
- 人工介入只允許存在於 `dispute`、`emergency`、`compliance exception`，不可把客服或人工審核寫成主流程 fallback
- Gate 內若有文件、命名、router、placeholder、DB spec 漂移，必須在該 Gate 內一併收斂，不可把漂移遞延到下一 Gate

### 4.2 Indexer 現況

目前 indexer 的正式用途為：

- 同步 `IdentityNFT.IdentityMinted`
- 將 `users.kyc_status` 更新為 `VERIFIED`

- 同步 `IdentityNFT.CredentialMinted`
- 回填 `user_credentials` 已啟用角色狀態

目前正式 worker 為：

- `modules/user/event_handler.go`
- `modules/credential/event_handler.go`

其餘 property / agency / listing / case 事件同步仍屬下一階段。

### 4.3 合約主線

目前身份主線以 `IdentityNFT.sol` 為準：

- tokenId `1`：NATURAL_PERSON
- tokenId `2`：OWNER
- tokenId `3`：TENANT
- tokenId `4`：AGENT

其中目前 live flow 已實際使用的是 `NATURAL_PERSON`。

## 六、下一階段開發方向

### 6.1 第二階段現況：Gate 1A 角色認證啟用閉環已交付

目標：讓已完成 KYC 的使用者可為 `OWNER / TENANT / AGENT` 提交證明、取得審核結果，並在自行決定後啟用角色 NFT。

#### 6.1.1 已交付項目

1. **三大角色正式頁面**
   - `/credential/owner`
   - `/credential/tenant`
   - `/credential/agent`
   - `IdentityCenterPage` 已改為正式顯示角色進度與啟用狀態

2. **智能 / 人工雙路審核**
   - 使用者可先走智能審核，再自行決定是否採用結果
   - 使用者也可改走人工審核
   - 通過 / 不通過結果都會回身份中心，不會直接替使用者做最終決定

3. **啟用後才 mint 角色 NFT**
   - 審核通過只會進入 `待啟用`
   - 只有使用者在前端主動點擊啟用時，才會 mint `OWNER / TENANT / AGENT` NFT
   - `CredentialMinted` 事件已同步回 `user_credentials`

4. **Admin 審核 API**
   - `GET /api/admin/credentials/pending`
   - `PUT /api/admin/credentials/:id/review`

#### 6.1.2 Gate 1A 的資料層分工

| Table | 用途 |
|-------|------|
| `credential_submissions` | 角色申請 write model，保存表單、OCR、審核路線、判定結果、啟用狀態 |
| `user_credentials` | 已啟用角色 read model，保存 `review_status`、`nft_token_id`、`tx_hash`、撤銷資訊 |

#### 6.1.3 第二階段剩餘重點：Gate 1B 權限切換

Gate 1A 已把角色申請與啟用做成閉環，但目前仍**沒有**把平台操作權限切到角色憑證。下一步 Gate 1B 才處理：

- 刊登房源從 `KYC VERIFIED` 切到 `OWNER` 憑證
- 預約看房從 `KYC VERIFIED` 切到 `TENANT` / 指定角色規則
- 仲介能力切到 `AGENT` 憑證
- 權限矩陣與 fallback 規則正式收斂

### 6.2 後續 phase（第三階段以後）

- property / agency 正式模組化
- 鏈上 CaseTracker 接線（LOCKED → SIGNING → CLOSED 狀態鏈上驅動）
- ListingStakeVault 上架押金機制
- 房屋平台完整鏈上狀態機
- WebSocket / 更完整通知體驗（目前 repo 尚無 hub / router）
- 若商業與法規條件成熟，再評估導入 TWID 或第三方 KYC provider
