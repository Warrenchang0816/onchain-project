## 核心原則

| 原則 | 說明 |
|------|------|
| **On-chain First** | 合約是狀態唯一真相來源（Source of Truth） |
| **正確資料流向** | `區塊鏈 ↔ 前端（MetaMask）↔ 後端（API）↔ 資料庫` |
| **後端無錢包** | 後端不持有私鑰，不呼叫合約；所有合約寫入走 MetaMask |
| **DB 為輔助層** | 索引、查詢優化、原始文件存放、後台審核 |
| **KYC 不自建** | 採第三方身份驗證（第一階段：TWID），不存原始個資 |
| **第一階段：租屋** | 先跑通租屋市場，買賣為第二階段 |

## 零、系統架構

```
                  ┌─────────────────────────────┐
                  │    Ethereum / Sepolia        │
                  │   （Source of Truth）         │
                  └──────┬──────────────┬────────┘
            emit events  │              │ send tx（MetaMask）
                  ┌──────▼──────┐  ┌───▼────────────────────┐
                  │ Go Indexer  │  │    前端 React           │
                  │（只讀 RPC） │  │  （用戶 MetaMask 操作）  │
                  └──────┬──────┘  └───┬────────────────────┘
                         │             │ REST API / WebSocket
                  ┌──────▼─────────────▼────────┐
                  │         Go-service           │
                  │  API + Indexer + OPERATOR key│
                  │  + Hash + Preflight + Notifier│
                  └──────────────┬──────────────┘
                                 │ SQL
                  ┌──────────────▼──────────────┐
                  │         PostgreSQL           │
                  │   （索引層、查詢、文件存放）  │
                  └─────────────────────────────┘
```

**三條核心線：**
```
用戶操作線：  用戶 → MetaMask → 合約（registerProperty, createListing 等）
平台自動線：  業務條件達成 → Go OPERATOR key → 合約（mint, verify, openCase 等）
追溯線：     鏈上 event → Go Indexer → DB（所有狀態皆可在鏈上驗證）
```

### 誰呼叫合約

| 呼叫者 | 身份 | 負責的合約函式 |
|--------|------|--------------|
| **用戶前端** | 屋主 / 刊登者 MetaMask | `registerProperty`, `addOwner`, `setPrimaryOperator`, `updateDisclosureHash`, `grantAuthorization`（含合約條款）, `revokeAuthorization`（未媒合時）, `createListing`（ETH）, `createListingWithToken`（USDC/USDT）, `payServiceFeeETH`, `payServiceFeeToken` |
| **Go-service（OPERATOR key）** | 平台自動化執行 | `IdentityNFT.mint/rebind/revoke`, `verifyOwner`, `verifyProperty`, `releaseStake`, `slashStake`, `forceRevoke`, `lockForMutualConsent`（案件進 MATCHED 時）, `revokeWithConsent`（雙方合意解約）, `openCase`（含仲介足跡）, `advanceCase`, `cancelCase`, `disputeCase`, `resolveDispute` |
| **AI Agent**（Phase 2） | 物件審核後輸出結果 | 不直接呼叫合約，輸出審核結論給 Go OPERATOR 執行 |

> **人工介入原則：極小化**。平台人工只處理申訴 / 客服爭議，不介入正常業務流程。

### 後端如何同步鏈上狀態到 DB

```
合約 emit event
    → Go Indexer（只讀，無錢包）
        → 更新 DB 對應欄位（*_tx_hash, status, token_id…）
        → Notifier 推送 WebSocket 通知給前端
```

## 二、資料庫 Schema（infra/init/02-house-platform.sql）

### 資料表關係

```
users (KYC + 自然人畫像)
  ├─ agent_profiles (仲介業務檔案，role=AGENT 專用)
  └─ property_owners ─┐
                       ├─ properties (物件)
                       │    ├─ property_authorizations (委託授權 + 合約條款)
                       │    └─ listings (刊登 + 看屋時段)
                       │         ├─ listing_views (訪客瀏覽 + 畫像快照)
                       │         └─ cases (案件 + 仲介引入足跡)
                       │              └─ case_events (事件日誌)
                       └─ (共同持有人)

tenant_demands        (租客需求清單)
indexer_checkpoints   (Indexer 斷點續傳)
processed_events      (Re-org 防護 + 冪等)
```

### 各表重點設計

| 資料表 | 關鍵欄位 | 說明 |
|--------|---------|------|
| `users` | `kyc_reference_id` UNIQUE | TWID 回傳的識別碼，一人一帳號；Go OPERATOR 自動 mint 後寫入 |
| `users` | `identity_hash` | SHA-256(TWID data)，不可還原個資 |
| `users` | `role` | USER / OWNER / AGENT（無 ADMIN，平台操作走 OPERATOR key） |
| `users` | `occupation`, `income_range`, `family_status`, `household_size` | 自然人畫像（KYC 後自填，用於媒合推薦與訪客統計） |
| `users` | `profile_completed` | TRUE 後才納入統計與推播 |
| `users` | `agent_*` | 仲介執照、公司、品牌（AGENT 專用） |
| `agent_profiles` | `license_verified`, `specializations`, `service_areas` | 仲介業務檔案，執照審核後才能接受委託 |
| `properties` | `deed_hash` UNIQUE | SHA-256(產權文件)，防重複登記 |
| `properties` | `disclosure_*` | 揭露義務欄位（含抵押/查封），未完成不可上架 |
| `properties` | `bedroom_count`, `bathroom_count`, `appliances`, `pet_allowed` | 格局、家電、寵物政策等出租頁面細項 |
| `properties` | `utility_type`, `utility_notes` | 水電費計算方式 |
| `property_owners` | `share_bps`, `is_primary_operator` | 共同持有 + 主操作人 |
| `property_owners` | `owner_verified` | Go OPERATOR key 呼叫 verifyOwner 後設 true（強屋主驗證） |
| `property_authorizations` | `service_fee_bps`, `mandate_duration_days`, `penalty_amount` | 委託合約核心條款，授權時確認 |
| `property_authorizations` | `mutual_consent_required` | FALSE=可單方解約；TRUE=案件 MATCHED 後鎖定（防過河拆橋） |
| `listings` | `viewing_schedule` JSONB | 可線下看屋時段 |
| `listings` | `current_rental_status` | VACANT / OCCUPIED / NOTICE_GIVEN |
| `listing_views` | `viewer_user_id` + 畫像快照 | 訪客瀏覽紀錄，供出租頁面族群統計 |
| `tenant_demands` | `preferred_areas`, `max_rent`, `pet_required` | 租客需求條件，供推播與屋主主動媒合 |
| `tenant_demands` | `open_to_agent_contact` | FALSE=預設不開放仲介聯絡 |
| `cases` | `introduced_by_agent_wallet` | 仲介引入足跡，防過河拆橋申訴依據 |
| `cases` | `disputed_at`, `disputed_tx_hash` | 糾紛時間戳與鏈上 tx |
| `cases` | 個別 `*_at` + `*_tx_hash` | 每個狀態有對應時間戳與鏈上 tx |
| `case_events` | append-only | 永不刪改，完整事件歷史（含 DISPUTED / DISPUTE_RESOLVED） |
| `indexer_checkpoints` | `last_processed_block` | Indexer 重啟後從此 block + 1 繼續 |
| `processed_events` | `(tx_hash, log_index)` PK | 防重複處理，Re-org 保護 |

## 四、後端架構（每個功能的檔案清單）

> 架構採 **platform / db / modules / bootstrap** 四層分離  
> 詳見 `docs/架構設計書.md § 二、Go-service 內部架構`

每個功能的檔案分布在兩個位置：
- **`internal/db/`**：model + repository（以 table 為單位）
- **`internal/modules/xxx/`**：dto + handler + service + event_handler（以業務為單位）

### 功能 1：KYC / User（TWID 整合）

```
internal/db/model/user_model.go
internal/db/repository/user_repo.go

internal/modules/user/dto.go
internal/modules/user/handler.go
internal/modules/user/service.go
internal/modules/user/event_handler.go     ← IdentityNFT events
```

API：
```
GET    /api/kyc/me               # 取得自己的 KYC 狀態（需登入）
POST   /api/kyc/twid/start       # 取得 TWID 驗證跳轉 URL（需登入）
POST   /api/kyc/twid/callback    # TWID 驗證完成後的 callback（由 TWID 呼叫）
```

**TWID 流程（全自動，無需人工）：**
1. `POST /kyc/twid/start` → 後端向 TWID 取得驗證 URL → 回傳前端
2. 前端跳轉 TWID → 使用者完成身份驗證
3. TWID 呼叫 `POST /kyc/twid/callback` → 後端驗 HMAC，取得 `kyc_reference_id`，驗證一人一帳號
4. **Go OPERATOR key 自動呼叫** `IdentityNFT.mint()` → SBT 鑄造到使用者錢包
5. Indexer 收到 `IdentityMinted` event → DB 更新 `kyc_status = VERIFIED`、`identity_nft_token_id`

### 功能 2：Property + PropertyOwner（含強屋主驗證 + 共同持有）

```
internal/db/model/property_model.go
internal/db/model/property_owner_model.go
internal/db/repository/property_repo.go
internal/db/repository/property_owner_repo.go

internal/modules/property/dto.go
internal/modules/property/handler.go
internal/modules/property/service.go
internal/modules/property/event_handler.go  ← PropertyRegistry events
```

API：
```
GET    /api/properties                 # 公開物件列表（VERIFIED）
GET    /api/properties/:id             # 物件詳情（含 owners、disclosure）
POST   /api/properties/upload-deed     # 上傳產權文件 → 計算 deed_hash（Preflight）
POST   /api/properties                 # 登記物件（前端接著送 registerProperty tx）
PUT    /api/properties/:id/disclosure  # 更新現況說明（前端接著送 updateDisclosureHash tx）
POST   /api/properties/:id/owners      # 新增共同持有人（前端接著送 addOwner tx）
GET    /api/preflight/register-property  # Preflight 驗證（送 tx 前呼叫）
```

自動審核流程：
```
MVP：Go 完成基本規則驗證（格式、hash 唯一、KYC 狀態）後 → OPERATOR key 自動執行 verifyOwner + verifyProperty
Phase 2：AI Agent 分析產權文件 → 輸出信心分數 → Go OPERATOR key 執行
```

**揭露義務規則：**
- 前端填寫 4 個揭露欄位 → 後端計算 `disclosure_hash`
- 前端 MetaMask 送 `updateDisclosureHash` tx
- Indexer 同步後 `disclosure_completed = true`
- **Preflight 刊登前強制驗證** `disclosure_completed = true`

### 功能 3：Authorization（Agency 委託授權）

```
internal/db/model/authorization_model.go
internal/db/repository/authorization_repo.go

internal/modules/agency/dto.go
internal/modules/agency/handler.go
internal/modules/agency/service.go
internal/modules/agency/event_handler.go    ← AgencyRegistry events
```

API：
```
GET    /api/properties/:id/authorizations            # 列出此物件的授權
POST   /api/properties/:id/authorizations            # Preflight + 引導 MetaMask grantAuthorization（含合約條款）
DELETE /api/properties/:id/authorizations/:authId    # Preflight + 引導 MetaMask revokeAuthorization（未媒合時）
```

> **revokeAuthorization Preflight 規則：**
> - 若 `mutual_consent_required = TRUE`（已有進行中案件）→ 拒絕，引導走雙方合意流程
> - 若 `mutual_consent_required = FALSE` → 允許屋主單方撤銷

### 功能 4：Listing（上架 + 刊登市場 + 押金）

```
internal/db/model/listing_model.go
internal/db/repository/listing_repo.go

internal/modules/listing/dto.go
internal/modules/listing/handler.go
internal/modules/listing/service.go
internal/modules/listing/event_handler.go   ← ListingStakeVault events
```

API：
```
GET    /api/listings              # 公開市場看板（ACTIVE，含 reservation_count、view_count）
GET    /api/listings/:id          # 刊登詳情
POST   /api/listings/preflight    # 刊登前 Preflight（驗證 VERIFIED + disclosure_completed）
POST   /api/listings              # 建立刊登記錄（前端接著送 createListing（ETH）或 createListingWithToken（USDC/USDT））
PUT    /api/listings/:id/view     # 增加 view_count（匿名可呼叫）
```

**Preflight 規則（刊登）：**
```
✓ 呼叫者 kyc_status == VERIFIED
✓ property.verification_status == VERIFIED
✓ property.disclosure_completed == TRUE
✓ 若呼叫者非 primaryOperator → property_authorizations 中需有 ACTIVE 授權
```

### 功能 5：Case + CaseEvent（案件追蹤）

```
internal/db/model/case_model.go
internal/db/model/case_event_model.go
internal/db/repository/case_repo.go
internal/db/repository/case_event_repo.go

internal/modules/case/dto.go
internal/modules/case/handler.go
internal/modules/case/service.go

```

API：
```
GET    /api/cases                  # 我的案件列表（需登入）
GET    /api/cases/:id              # 案件詳情（含 case_events 時間軸）
POST   /api/cases/intent           # 提交意向（DB 記錄，WebSocket 通知屋主）

```

```
取消條件觸發 → cancelCase() 自動執行
```

## 六、角色與業務規則

### 前端 / 後端 Guard

```
未登入                         → 可讀 /api/listings（公開市場）
已登入（SIWE）                 → 可提交 KYC 申請
KYC VERIFIED                  → 可登記物件、提交意向
KYC VERIFIED + owner_verified  → 可揭露現況、授權仲介、刊登
KYC VERIFIED + role=AGENT      → 持有授權後可刊登他人物件
```

### 核心業務驗證（Service Preflight 層）

| 操作 | 驗證條件 |
|------|---------|
| 提交 KYC | `kyc_reference_id` 不得已存在其他帳號（一人一帳號） |
| 登記物件 | 呼叫者 `kyc_status == VERIFIED`；`deed_hash` 不得重複 |
| 揭露現況 | 呼叫者為此物件的 `primaryOperator` |
| 授權仲介 | 呼叫者為此物件的 `primaryOperator`；物件已 VERIFIED |
| 刊登 | 物件 `verification_status == VERIFIED`；`disclosure_completed == TRUE`；非屋主刊登需有 ACTIVE 授權 |
| 案件狀態推進 | **Go OPERATOR key 自動執行**；Service 層依 `case_type` 驗證合法狀態轉換 |

### 合約寫入職責（最終定版）

| 情境 | 呼叫者 | 觸發方式 |
|------|--------|---------|
| KYC mint SBT | **Go-service（OPERATOR key）** | TWID callback 驗證通過 → 自動執行 |
| 物件登錄 | **用戶前端（MetaMask）** | 屋主主動操作 |
| 強屋主驗證 | **Go-service（OPERATOR key）** | MVP：基本規則；Phase 2：AI Agent 審核 → 自動執行 |
| 揭露義務上鏈 | **用戶前端（MetaMask）** | 屋主主動填寫 |
| 授權仲介 | **用戶前端（MetaMask）** | 屋主主動操作 |
| 撤銷授權 | **用戶前端（MetaMask）** | 屋主主動操作 |
| **刊登押金（ETH）** | **用戶前端（MetaMask）** | `createListing` 直接鎖倉 ETH |
| **刊登押金（USDC/USDT）** | **用戶前端（MetaMask）** | `createListingWithToken` 直接鎖倉穩定幣（部署時固定支援） |

| 案件開立 / 推進 | **Go-service（OPERATOR key）** | 業務條件達成 → 自動執行 |

> **用戶 MetaMask 只做用戶自己的操作**。平台操作由 Go OPERATOR key 自動執行，全程鏈上可追溯，無人工介入正常業務流程。

## 八、狀態機（Final v1）

### 租屋（RENT）
```
OPEN → MATCHED → SIGN → CLOSED
  └──────────────────────→ CANCELLED
  └──────────────────────→ DISPUTED → resolveDispute → CLOSED / CANCELLED
```

### 買賣（SALE）
```
OPEN → MATCHED → SIGN → BANKING → CLOSED
  └────────────────────────────────→ CANCELLED
  └────────────────────────────────→ DISPUTED → resolveDispute → CLOSED / CANCELLED
```

> MATCHED 觸發：若有仲介委託，Go OPERATOR key 自動呼叫 `AgencyRegistry.lockForMutualConsent()`

詳見 `docs/架構設計書.md § 六、Case 狀態機`

## 九、KYC 身份驗證架構（TWID 整合）

### 設計定調

> KYC 不該是平台自己做的功能，而是平台接入的信任來源。

平台不自建 KYC，採第三方身份驗證服務，第一階段使用 **TWID**。

### 完整流程（全自動，無人工介入）

```
使用者（MetaMask）
    ↓ SIWE 登入
後端建立 session
    ↓ POST /api/kyc/twid/start
後端向 TWID 取得驗證 URL
    ↓ 回傳前端
前端跳轉 TWID
    ↓ 使用者完成身份證 + 手機 + 活體驗證
TWID callback → POST /api/kyc/twid/callback（後端）
    ↓ 後端驗 HMAC，取得 kyc_reference_id，驗一人一帳號
Go OPERATOR key 自動呼叫 IdentityNFT.mint()
    ↓ SBT 鑄造到使用者錢包（鏈上記錄，公開可查）
Indexer 收到 IdentityMinted event
    ↓ DB 更新 kyc_status = VERIFIED、identity_nft_token_id
使用者取得完整平台權限
```

### 資料儲存原則

| 資料 | 存放位置 | 說明 |
|------|---------|------|
| 身分證影像 | **不存** | 由 TWID 持有，平台不碰 |
| 姓名 / 身分證號 | **不存** | 不儲存原始個資 |
| `kyc_reference_id` | DB（UNIQUE） | TWID 回傳的識別碼，確保一人一帳號 |
| `identity_hash` | DB | 平台計算的 hash，不可還原 |
| SBT tokenId | DB + 鏈上 | `IdentityNFT.sol` 的 tokenId（Indexer 同步） |

### 換錢包流程（Wallet Rebind）

1. 使用者重新透過 TWID 驗證（確認仍是同一人）
2. 後端取得同一 `kyc_reference_id`
3. **Go OPERATOR key** 呼叫 `IdentityNFT.rebind(referenceId, newWallet)`
4. 舊錢包 SBT 自動 burn，新錢包 mint 新 SBT
5. Indexer 同步 → DB 更新 `wallet_address` 對應記錄

## 十、現有 Task 模組的兼容策略

### 現況評估

原任務追蹤模組（`TaskRewardVault`）採用的是**舊架構**：後端持有 OPERATOR 私鑰並直接呼叫合約。這與新的 On-chain First 架構不符，但：

- 功能本身運作正常，是公版 boilerplate 的核心範例
- 重構代價高，且不在現階段目標內
- 兩套模組邊界清晰，不互相污染

### 策略：共存，不改，不擴

| 決策 | 原因 |
|------|------|
| 保留現有 task 模組原樣 | 不破壞已驗證的功能 |
| 新房屋平台模組嚴格遵守新架構 | 兩套並存，邊界隔離 |
| 未來若要升級 task 模組 | 按新架構重寫，當作 On-chain First 練習 |

### 新舊架構對比

| 面向 | 舊（Task 模組） | 新（房屋平台）|
|------|--------------|-------------|
| 合約呼叫者 | Go-service（持私鑰） | 前端 MetaMask |
| 鏈上狀態同步 | 後端直接呼叫後知道結果 | Indexer 訂閱 events |
| DB 狀態更新 | 合約 tx 完成後立即更新 | Indexer 非同步更新（~12s 延遲）|
| Re-org 保護 | 無 | processed_events + confirmation blocks |
