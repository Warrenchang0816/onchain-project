# 第六階段架構定稿：可信房屋媒合平台（On-chain First）

## 文件目的

本文件用於確認第六階段之產品方向、系統邏輯、角色模型、交易路線、狀態流程與開發原則，作為後續資料庫設計、智能合約設計與功能開發的共同基準。

本版本已明確定調：

- 平台本質不是一般房屋資訊網站
- 平台核心是「可信房屋媒合市場」
- 系統從一開始即採用 **On-chain First** 設計思維
- 鏈上狀態為核心真相來源（Source of Truth）
- DB 作為索引、查詢優化、文件存放與後台審核輔助層

---

# 一、產品核心定位

## 1.1 專案本質

本平台的核心，不是讓找房更方便，而是讓房屋市場更可信。

平台不是單純刊登資訊的工具，而是一套透過技術規則建立信任、權限與責任的市場系統。

---

## 1.2 核心使命

建立一個讓：

- 屋主可以安心自售、自租
- 租客與買方可以接觸到更真實、更完整的物件
- 仲介回歸服務業本質，而非資訊壟斷者

的可信房屋市場。

---

## 1.3 核心價值主張

### 對屋主
- 屋主可以自行出租、自行出售
- 屋主掌握物件完整控制權
- 未經授權，任何人不得操作物件
- 平台作為可被信任的房屋品牌，協助屋主安全進入市場

### 對租客 / 買方
- 減少假物件與資訊不實
- 降低搜尋成本與被誤導風險
- 透過平台機制取得更可信的物件資訊

### 對仲介
- 仲介不被消滅，但角色被重構
- 仲介必須回歸服務本質
- 仲介作為線下輔佐者、專業服務者與人脈資源提供者存在

---

# 二、核心產品理念

## 2.1 主軸定義

本平台是一個：

> **屋主友善、可追溯、可驗證、能清掃不良仲介的信任平台**

其本質是用系統規則取代傳統市場中模糊且容易被濫用的信任關係。

---

## 2.2 市場問題對應

本平台欲改善之問題包括：

- 假物件
- 重複刊登
- 地址與內容不實
- 仲介未經授權操作物件
- 屋主失去資產控制權
- 租客被迫自行查找大量無效資訊
- 租客遭遇不良仲介與不實案件
- 市場缺乏可追溯的完整流程資料

---

## 2.3 核心方法

平台透過以下機制建立可信度：

- KYC 實名驗證
- 強屋主驗證
- 物件憑證
- 授權控制
- 物件揭露義務
- Stake 行為成本
- Case 全流程狀態追蹤
- 鏈上公開狀態存證

---

# 三、On-chain First 架構原則

## 3.1 核心原則

本專案從一開始即以鏈上設計為基礎，而非先做中心化資料庫、未來再考慮上鏈。

系統中涉及信任、權限、狀態與責任之核心資料，皆需以鏈上狀態為準。

---

## 3.2 鏈上與鏈下分工

### 鏈上負責
鏈上保存具備「真相來源」地位的核心資料與狀態，包括：

- 身分憑證
- 物件憑證
- 物件持有人關係
- 物件揭露資訊的 hash 與版本
- 授權狀態
- 上架狀態
- 媒合狀態
- 案件狀態
- Stake / Slash 狀態
- 同意書狀態
- 路線選擇狀態

### 鏈下負責
鏈下 DB / Storage 僅負責：

- 原始證件影像
- 房屋照片
- 權狀 / 謄本掃描檔
- KYC 原始資料
- AI / OCR / 人工審核結果
- 索引與查詢優化
- 鏈上同步紀錄
- 後台管理資訊

---

## 3.3 公開透明原則

系統必須兼顧：
- 公開透明
- 可追溯
- 個資保護

因此：

- 文字型關鍵狀態可上鏈
- 敏感身分資料不直接公開上鏈
- 原始文件不直接上鏈
- 以 hash 與 metadata 方式鏈上存證

---

# 五、角色與身份模型

## 5.1 基礎角色

系統中的人（Person）與角色（Role）需分開理解。

同一個人可同時具有不同角色。

### 角色類型
- USER：一般使用者
- OWNER：經強驗證的資產持有人
- AGENT：經 KYC 且具職業標記的仲介

---

## 5.2 KYC

KYC 代表：
- 你是誰
- 你是可追溯的真人
- 你可被系統納入責任與信用體系

KYC 本身不等於屋主資格。

### 自然人畫像（KYC 後自填）

KYC 完成後，使用者可自行填寫畫像資料，用於：
- 媒合推薦（系統依畫像條件推播物件 / 租客）
- 出租頁面訪客群組統計（屋主可看到哪些族群在瀏覽物件）

畫像欄位（自申報，不與身分證資訊綁定）：
- 職業（occupation）
- 月收入區間（income_range）
- 家庭狀況（family_status：SINGLE / COUPLE / FAMILY_WITH_CHILD / MULTI_GEN / OTHER）
- 同住人數（household_size）

畫像填寫完成後（profile_completed = TRUE）才納入統計與推播。

---

## 5.3 強屋主驗證

強屋主驗證代表：
- 此人不只是經過身份驗證
- 更被確認對某一房屋具有合法持有或操作資格

因此系統需區分：

- User Identity
- Owner Credential

---

## 5.4 仲介資格

仲介必須具備：

- KYC（TWID 實名驗證）
- 職業標記（role = AGENT）
- 不動產營業員 / 經紀人執照號碼（agent_license_no）
- 公司 / 品牌欄位（agent_company / agent_brand）
- 執照由後台審核（license_verified）

仲介業務檔案（agent_profiles 表）：
- 自我介紹（bio）
- 執業年資（years_of_experience）
- 專長項目（specializations：RENT / SALE / LUXURY / COMMERCIAL）
- 服務區域（service_areas）
- 是否接受新委託（is_accepting_mandates）
- 業務頁面公開狀態（profile_visible）

成交紀錄 / 接受委託件數：從 cases / property_authorizations 表動態查詢，不另建表。

第一階段先確保功能能運作，後續再依台灣現行業界標準補強商業邏輯與法規細節。

---

# 六、物件模型與共同持有

## 6.1 Property 概念

Property 不是單純貼文，而是系統中的核心資產單位。

一個 Property 應具備：
- 唯一物件識別
- 與真實房屋對應
- 可驗證的權屬關係
- 可追蹤的揭露與授權狀態

---

## 6.2 共同持有設計

現實世界中，房屋常有共同持有狀況，例如：
- 夫妻共有
- 家族共有
- 持分共有

因此 Property 不應綁死單一 owner，而應設計為：

- 一個 Property
- 對應多個 PropertyOwner

---

## 6.3 MVP 建議做法

第一階段先支援：

- 一個物件可有多位持有人
- 記錄每位持有人的持分比例
- 指定一位主操作人（Primary Operator）

### 原因
若一開始即要求所有持有人共同多簽，將使流程過度複雜，難以先跑通 MVP。

因此第一階段先讓系統可運作，後續再評估升級為多簽機制。

---

# 七、物件揭露義務（Disclosure）

## 7.1 原則

物件完整性必須成為平台的基本規則，而非選填參考資訊。

屋主在認證物件時，應負有揭露義務。

---

## 7.2 第一階段強制揭露項目

至少應包含：

- 是否為凶宅
- 是否有漏水問題
- 是否存在重大糾紛
- 是否有人居住 / 租約中
- 是否設有抵押（mortgage）
- 是否有查封 / 假扣押（lien）
- 其他重大瑕疵與補充說明

---

## 7.3 規則

- 未完成揭露，不可上架
- 揭露不實，可作為後續檢舉與處置依據

---

# 八、信任層核心模組

## 8.1 KYC（身份）
目的：
- 一人一身份
- 防止假帳號與多帳號亂入

---

## 8.2 Property（物件）
目的：
- 先驗證物件，才能進市場
- 建立房屋與數位憑證關係

---

## 8.3 Agency（授權）
目的：
- 屋主掌握最終控制權
- 仲介必須被授權才能操作

委託合約核心條款（授權時雙方確認，鏈上存證）：
- 服務費率（service_fee_bps）
- 委託期限（mandate_duration_days）
- 違約金（penalty_amount）

解約規則：
- 尚未進入媒合（no active case）：屋主可單方撤銷
- 已進入媒合（case status ≥ MATCHED）：平台自動鎖定（lockForMutualConsent），需雙方合意才可解約
- 防過河拆橋：案件引入仲介足跡（introducedByAgent）鏈上存證，屋主解約後私下成交可觸發 slash 申訴

---

## 8.4 Stake（行為成本）
目的：
- 將行為與成本綁定
- 提高違規與說謊成本
- 過濾垃圾資訊與惡意互動

---

## 8.5 Case Tracking（案件追蹤）
目的：
- 將媒合與交易過程轉為可追蹤的數位狀態
- 建立後續市場資料與信用基礎

---

# 九、交易與媒合路線

## 9.1 第一階段與第二階段主軸

### 第一階段：租屋
- 先做租屋市場
- 主流程以平台媒合、線下完成為主
- 平台持續追蹤案件到結案

### 第二階段：買賣
- 再做中古屋買賣
- 平台仍以媒合與流程追蹤為主
- 預留更進階路線

---

## 9.2 四條路線

### 路線一：傳統媒合
- 當前主線
- 線下完成交易
- 平台負責媒合與狀態追蹤

### 路線二：RWA
- 第二階段選項
- 將物件做資產映射
- 第一階段不實作

### 路線三：鏈上交易
- 保留路線
- 依物件上鏈程度與同意書決定是否進入鏈上交易
- 第一階段不實作，但設計需預留

### 路線四：AI 仲介
- 保留路線
- 由平台 AI 代理部分媒合與流程工作
- 第一階段不實作，但設計需預留

---

# 十、案件狀態機（Final v1）

## 10.1 租屋流程

### 狀態
- OPEN
- MATCHED
- SIGN
- CLOSED
- CANCELLED（任何非終止狀態）
- DISPUTED（任何非終止狀態，需後台介入）

### 說明
#### OPEN
- 物件已上架
- 顯示目前有幾個預約
- 顯示目前有幾個看過

#### MATCHED
- 屋主收訂金
- 案件已達成媒合
- 觸發 AgencyRegistry.lockForMutualConsent（若有仲介委託）

#### SIGN
- 簽約流程中

#### CLOSED
- 案件成功結束

#### DISPUTED
- 任何階段均可進入
- 後台介入處理
- 解決後導向 CLOSED 或 CANCELLED

---

## 10.2 買賣流程

### 狀態
- OPEN
- MATCHED
- SIGN
- BANKING
- CLOSED
- CANCELLED（任何非終止狀態）
- DISPUTED（任何非終止狀態，需後台介入）

### 說明
#### OPEN
- 物件已上架
- 顯示目前有幾個預約
- 顯示目前有幾個看過

#### MATCHED
- 屋主收斡旋
- 進入簽約前成交確認階段
- 觸發 AgencyRegistry.lockForMutualConsent（若有仲介委託）

#### SIGN
- 簽約流程中
- 主要由代書負責

#### BANKING
- 銀行流程進行中

#### CLOSED
- 案件成功結束

#### DISPUTED
- 任何階段均可進入
- 後台介入處理
- 解決後導向 CLOSED 或 CANCELLED

---

# 十一、憑證設計原則

## 11.1 身分憑證
- 採 NFT / SBT 概念
- 作為身份與角色驗證之鏈上憑證

## 11.2 物件憑證
- 一個物件需有對應的鏈上憑證
- 作為物件可被操作、可被揭露、可被追蹤的前提

## 11.3 憑證更新
憑證更新不是更換主體，而是：

- 同一主體下的 metadata 更新
- 狀態版本更新
- 事件紀錄保留

例如：
- 新增揭露資訊
- 補件
- 狀態變更
- 持有人資訊修正

# 十二、經濟模型

## 12.1 Credit
- 作為新手與初期互動的門檻降低工具

## 12.2 Stake
- 作為市場治理工具

## 12.3 Fee
- 作為平台營收來源
- 用於支撐平台商業模式

# 十六、技術架構實作定稿

> 本節為開發過程中確立的技術決策，補充說明第三章 On-chain First 的具體實作方式。  
> 詳細技術規格見 `docs/架構設計書.md`。

## 16.1 架構模式名稱

本平台採用：**On-chain Event Sourcing + CQRS + Indexer Pattern**

| 模式 | 對應 |
|------|------|
| Event Sourcing | 鏈上 events 是唯一狀態來源，DB 是派生物 |
| CQRS | 寫入走 MetaMask → 合約；讀取走 Go API → DB |
| Indexer Pattern | Go goroutine 訂閱鏈上 events，同步至 PostgreSQL |

---

## 16.2 呼叫合約的責任切分

系統中只有兩種角色可以呼叫合約，沒有人工 Admin：

**用戶（MetaMask）**：用戶自己的操作
- `registerProperty`、`updateDisclosureHash`、`grantAuthorization`、`revokeAuthorization`
- `createListing`（ETH 鎖倉）、`createListingWithToken`（USDC / USDT 鎖倉）
- `payServiceFeeETH`、`payServiceFeeToken`（案件雙方直接上鏈支付服務費，透明可追溯）

**Go-service（OPERATOR key）**：平台自動化操作
- `IdentityNFT.mint()`：TWID callback 驗證通過後自動執行
- `verifyOwner` / `verifyProperty`：MVP 基本規則審核；Phase 2 AI Agent 審核後執行
- `openCase` / `advanceCase` / `cancelCase`：業務條件達成後自動執行
- `releaseStake` / `slashStake`：結案或違規條件達成後自動執行

> Go OPERATOR key 是平台的自動化行政鑰匙，不代替用戶操作，不接觸用戶資金。

### 合約角色設計原則

合約中使用四個語意明確的角色，**讀名稱即知職責，不存在歧義**：

| 角色 | 持有者 | 職責 |
|------|--------|------|
| `OPERATOR_ROLE` | Go-service 環境變數 | 平台自動化業務操作 |
| `TREASURY_ROLE` | 財務獨立錢包 | 提領累積服務費與沒收押金 |
| `EMERGENCY_ROLE` | 冷錢包（部署者） | 緊急全額提領（合約升級 / 意外情境） |
| `DEFAULT_ADMIN_ROLE` | 部署者（OZ 內部） | OZ 角色管理機制，不綁任何業務函式 |

### 穩定幣支援

`ListingStakeVault` 在部署時固定支援 ETH + USDC + USDT（地址傳 `address(0)` 代表不支援）：
```
部署時傳入：USDC_ADDRESS、USDT_ADDRESS（address(0) = 不支援）
合約內以 immutable 儲存，部署後不可更改
前端透過 supportedTokens() 查詢支援幣種與 symbol
```

---

## 16.3 人工介入原則

**極小化人工介入**：正常業務流程全部自動化，人工只處理例外。

| 情境 | 處理方式 |
|------|---------|
| KYC 驗證 | TWID 第三方 → Go 自動 mint，無人工 |
| 物件審核 | MVP：規則自動審核；Phase 2：AI Agent |
| 案件推進 | 業務條件達成自動觸發 |
| 申訴 / 客服爭議 | 客服後台介入（不在第一階段範圍內） |

---

## 16.4 AI Agent 物件審核（Phase 2）

平台專屬 AI Agent，餵入台灣不動產相關文件後進行機器學習：
- 產權狀、謄本辨識
- 所有人姓名與 KYC 身份比對
- 輸出信心分數 → Go OPERATOR key 執行 verifyProperty
- 信心分數不足 → 自動通知用戶補件，不需人工

> 架構設計已預留 AI Agent 接口，Phase 1 以規則審核佔位，Phase 2 替換即可。

## 16.5 Go-service 在 Web3 架構中的角色

Go-service 是平台的中樞，同時承擔：

| 角色 | 說明 |
|------|------|
| **REST API** | 提供前端查詢與互動接口 |
| **Indexer** | 訂閱鏈上 events，同步狀態到 DB |
| **OPERATOR** | 持 OPERATOR key，自動執行平台合約操作 |
| **Hash Server** | 計算 deed_hash、disclosure_hash、identity_hash |
| **Preflight** | tx 送出前驗證業務規則，避免浪費 gas |
| **Notifier** | WebSocket 即時推送鏈上事件給前端 |
| **TWID Client** | 呼叫 TWID API、驗證 HMAC callback |
