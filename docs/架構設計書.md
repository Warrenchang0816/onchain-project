## 一、整體架構原則

### 1.1 設計原則

- 以 **Ethereum / Sepolia 與鏈上事件** 作為鏈上狀態的真相來源（Source of Truth）。
- 以 **PostgreSQL** 作為索引層、查詢快取、off-chain workflow 與文件中繼層。
- 前端不可信任「交易已送出」本身；正式展示狀態以 Go service 讀出的 DB read model 為準，若與鏈上衝突，最終仍以鏈上事件 / receipt 裁定。
- 所有正式功能遵守開發順序：
  1. DB schema
  2. 合約與事件
  3. Go service / indexer / signer
  4. React `api` 與頁面整合

### 1.2 四層建構邏輯

| 層級 | 定義 | 本專案對應 |
|------|------|------------|
| 基礎設施層 | 環境變數、DB、RPC、外部雲服務、檔案儲存 | `go-service/internal/platform/*`、`infra/*` |
| 鏈上合約層 | Solidity 合約、事件、角色權限、SBT / Vault | `task-reward-vault/contracts/*` |
| 服務橋接層 | Go API、session、KYC 流程、indexer、合約服務 | `go-service/internal/modules/*` |
| 用戶互動層 | React、路由、錢包、KYC wizard、身份中心 | `react-service/src/*` |

### 1.3 目前正式主線

目前專案已完成的正式主線是：

1. `/kyc` 自建 eKYC onboarding（含接續進度精確步驟復原）
2. `/login` 使用 `person_hash + wallet SIWE + password`
3. `/forgot-password` 忘記密碼 / 初次設定密碼（Email OTP 驗證後設定）
4. `/member` 身份中心顯示自然人 KYC / IdentityNFT 狀態
5. `/profile` 會員資料頁（Email / 手機 / 通訊地址 OTP 變更）
6. `/listings` 房源列表與詳情（KYC 驗證用戶才能刊登）
7. 預約看房系統（排隊、屋主鎖定洽談、解鎖、結案）

### 1.4 未來藍圖與現況邊界

以下方向仍屬專案長期藍圖，尚未實作：

- `property`：物件獨立登記
- `agency`：仲介授權管理
- `case`：鏈上 CaseTracker 媒合
- OWNER / TENANT / AGENT 完整角色申請閉環（目前屬第二階段主線，但尚未閉環）

`modules/task/` 已全面移除，`modules/listing/` 為目前正式房屋媒合骨架。

### 1.5 主幹 Gate Roadmap 與中程鏈上化順序

自 2026-04-22 起，平台主體後續開發正式採 gate-based roadmap 治理；上位治理文件為：

- `docs/superpowers/specs/2026-04-22-platform-mainline-gate-roadmap-design.md`

中程正式主幹順序固定為：

1. `Gate 0`：Repo 基線收斂
2. `Gate 1`：身分與角色閉環
3. `Gate 2`：Property On-chain
4. `Gate 3`：Agency On-chain
5. `Gate 4`：Case On-chain
6. `Gate 5`：Stake 正式上鏈

架構治理補充：

- 平台主幹目前採 `租屋主線`；買賣路線僅保留架構預留與各 Gate 的前置欄位，不作為當前主幹正式需求
- `OWNER 自營` 與 `OWNER -> AGENT 授權` 兩條路線並存，owner-first 是平台定位的一部分，不可被單一路徑取代
- 鏈上化順序必須遵守 `Property -> Agency -> Case -> Stake`，避免在物件主體與授權邊界未穩定前先把案件與質押主線上鏈
- 複雜條款全文、附件、影像與商務文件原則上留在 DB / storage，以 hash 或 snapshot 與鏈上狀態對照
- `WebSocket / notify / TWID / AI 審核` 屬支援軌，不作為主幹 Gate 驗收阻塞條件

### 2.2 各層職責

#### `platform/`

跨模組共用基礎能力，不承擔業務決策：

- `platform/auth/`
  - SIWE nonce
  - session cookie
  - auth middleware
- `platform/blockchain/`
  - RPC client
  - hash 工具
- `platform/indexer/`
  - checkpoint
  - processed events
  - worker 註冊與掃描
- `platform/storage/`
  - MinIO / S3-compatible 檔案儲存
- `platform/ocr/`
  - Google Vision OCR
- `platform/faceai/`
  - AWS Rekognition CompareFaces
- `platform/notify/`
  - Email OTP
  - SMS OTP

#### `db/model/` 與 `db/repository/`

目前 live schema 直接對應的主要 model / repo：

- `users`
- `kyc_sessions`
- `kyc_submissions`
- `otp_codes`
- `auth_nonce`
- `wallet_session`
- `user_credentials`
- `listings`
- `listing_appointments`
- `task_blockchain_logs`（blockchain log 足跡，由 `modules/logs` 查詢）
- `indexer_checkpoints`
- `processed_events`

#### `modules/`

目前正式模組：

- `modules/auth`
  - SIWE / session / login handler（`handler.go`、`login_handler.go`）
  - 密碼重設 / 初次設定密碼（`reset_password_handler.go`）
    - `POST /api/auth/reset-password/request-otp`
    - `POST /api/auth/reset-password/set-password`
- `modules/onboarding`
  - KYC-first onboarding
  - OTP / OCR / 自拍 / 錢包綁定 / 設密碼
  - resume 精確步驟計算（`calcResumeWizardStep`）
  - Email 已註冊但未設密碼 → `ErrEmailNotActivated` 導向重設密碼頁
- `modules/user`
  - `/api/kyc/me`、`/api/user/profile`
  - IdentityNFT 合約互動
  - IdentityNFT event handler
  - Email / 手機 / 通訊地址 OTP 變更
- `modules/listing`
  - 房源 CRUD（僅 KYC VERIFIED 用戶可刊登）
  - 上架 / 下架 / 結案 / 狀態機
  - 預約看房系統（排隊 / 確認 / 洽談鎖定）
  - 日費計算（NT$40/天）
- `modules/logs`
  - 鏈上足跡查詢 `/api/blockchain-logs`

### 2.3 跨模組規則

- `modules/auth` 不承擔 OCR、人臉比對、KYC 影像處理。
- `modules/onboarding` 管理 onboarding 期間的 session / OTP / KYC 狀態。
- `modules/user` 負責最終 KYC 狀態、IdentityNFT 與 credential 狀態查詢。
- `modules/listing` 所有刊登操作必須先確認 `users.kyc_status = VERIFIED`。
- `modules/logs` 只提供查詢，不寫入鏈上狀態。
- `platform/*` 只提供能力，不直接承擔業務決策。

## 三、啟動流程與鏈上同步

### 3.1 `bootstrap/wiring.go` 啟動流程

目前實際 wiring 順序如下（`bootstrap/wiring.go`）：

1. 初始化 PostgreSQL
2. 載入 `blockchain / eKYC` 設定
3. 若有 RPC，初始化只讀 eth client
4. 初始化所有 repositories（含 `listing_repo`、`appt_repo`）
5. 初始化 `logs` module
6. 初始化 `auth` module
7. 初始化 eKYC 平台服務（MinIO / Google Vision / AWS Rekognition）
8. 初始化 indexer，視 `IDENTITY_NFT_ADDRESS` 註冊 Identity worker
9. 初始化 `user` module
10. 初始化 notify services（Email / SMS）
11. 初始化 `onboarding` module
12. 初始化 `user handler` 與 `admin handler`
13. 初始化 `listing` module
14. 註冊 router

### 3.2 `platform/indexer/` + `modules/user/event_handler.go`

目前 indexer 的正式用途只有一個：  
**同步 `IdentityNFT.IdentityMinted` 事件，把自然人 KYC 狀態回寫到 `users`。**

#### 實際模式

- 採 **on-demand sync**，不是全系統多模組常駐 worker 全開。
- 目前正式註冊的 worker 只有：
  - `modules/user/event_handler.go` → `IdentityWorker`

> `listing` 目前的狀態機為純 DB，不接 indexer 事件。CaseTracker / ListingStakeVault 事件同步屬後續 phase。

#### 實際同步流程

1. Go 端呼叫 `IdentityNFT.mint()`
2. 需要確認鏈上狀態時，觸發 indexer 掃描
3. indexer 讀取 safe block
4. `IdentityWorker` 解析 `IdentityMinted`
5. `users.kyc_status = VERIFIED`
6. `users.identity_nft_token_id = tokenId`

#### 冪等與進度保護

- `indexer_checkpoints`
  - 記錄每個 contract worker 的最後處理 block
- `processed_events`
  - `(tx_hash, log_index)` 保證同一事件不會重複處理

> 先前文件中的 `modules/property/event_handler.go`、`modules/agency/event_handler.go`、`modules/listing/event_handler.go`、`modules/case/event_handler.go` 仍屬未來 phase，**目前 repo 內不存在，不應視為現況**。

### 3.3 `platform/notify/`

目前通知層的正式能力只有：

- Email OTP
- SMS OTP
- 開發模式下 log-only fallback

目前 **沒有** `platform/notifier/` WebSocket Hub，也沒有即時推播 router。  
如果未來需要，即時通知會是後續體驗層 enhancement，而不是目前 live architecture 的一部分。

## 四、完整資料流

### 4.1 平台資料流總覽（設計基準）

```text
┌──────────────────────────────────────────────────────────┐
│                   Ethereum / Sepolia                      │
│              （Source of Truth，鏈上狀態為準）             │
└────────────────────┬─────────────────┬────────────────────┘
                     │ emit events      │ send tx（MetaMask）
              ┌──────▼──────┐    ┌──────▼──────┐
              │ Go Indexer  │    │   前端       │
              │（訂閱 events）│    │（用戶 MetaMask）│
              └──────┬──────┘    └──────┬──────┘
                     │ 同步 DB          │ HTTPS / WSS
              ┌──────▼──────────────────▼──────┐
              │            Go-service           │
              │   REST API + WebSocket + Crypto │
              └──────────────┬─────────────────┘
                             │ SQL
              ┌──────────────▼─────────────────┐
              │          PostgreSQL             │
              │    （索引層、查詢快取、文件存放）  │
              └────────────────────────────────┘
```

這張圖是本平台的正式資料流基準，對應的是 **On-chain Event Sourcing + CQRS + Indexer Pattern**：

- 鏈上合約與 events 是鏈上狀態的最終裁定來源，DB 不是主真相來源，而是衍生 read model。
- 前端自己的鏈上寫入一律走 MetaMask；前端查詢、登入、KYC、會員資料與 off-chain 流程一律走 Go API。
- Go Indexer 負責把鏈上事件同步到 PostgreSQL，供 API 查詢、身份中心顯示、冪等保護與追蹤使用。
- PostgreSQL 除了鏈上 read model，也承載目前尚未 fully on-chain 化的流程資料，例如 `kyc_sessions`、`kyc_submissions`、`listings`、`listing_appointments`、`user_credentials`。
- 目前 live repo 已有 REST API、indexer 與鏈上同步；`WebSocket / WSS` 仍屬預留擴充位，尚未在現行 router 中正式接線。

### 4.2 KYC 驗證流程（目前 live flow）

目前專案 **不走 TWID callback 流程**。  
正式流程是自建 eKYC + KYC-first onboarding：

1. Email OTP
2. Phone OTP
3. 上傳身分證正反面
4. OCR 提取姓名 / 生日 / 地址 / 身分資訊
5. 使用者確認 OCR 資料
6. 上傳第二證件
7. 上傳自拍做人臉比對
8. 綁定唯一錢包並完成 SIWE 簽名
9. 建立 `users` + `kyc_submissions`
10. Go OPERATOR key 直接 mint `IdentityNFT` tokenId=1
11. indexer 同步後，`users.kyc_status = VERIFIED`

### 4.3 Listing 狀態機（目前 live flow）

```
DRAFT → (publish) → ACTIVE
ACTIVE → (lockNegotiation) → NEGOTIATING
NEGOTIATING → (unlockNegotiation) → ACTIVE
ACTIVE / NEGOTIATING → (close) → CLOSED
ACTIVE → (remove) → REMOVED
ACTIVE → (expires) → EXPIRED
```

狀態機目前以純 DB 驅動，鏈上 CaseTracker / ListingStakeVault 接線為後續 phase。

### 4.4 房屋平台後續主線（未來 phase）

後續平台會拆出以下正式領域：

- `property`：物件獨立登記與產權追蹤
- `agency`：仲介授權與主權管理
- `case`：鏈上 CaseTracker 媒合（LOCKED → SIGNING → CLOSED 鏈上接線）

這些模組尚未在 repo 內建立，不應描述成現況。

## 五、角色與權限

### 5.1 使用者角色

| 角色 | 取得方式 | 鏈上憑證 | 目前狀態 |
|------|---------|---------|---------|
| 未登入 | 無需 | 無 | 已存在 |
| USER | `person_hash + wallet SIWE + password` 登入 | 無 | 已存在 |
| KYC VERIFIED | 完成 onboarding + IdentityNFT minted | `IdentityNFT` tokenId=1 | 已存在 |
| OWNER | KYC VERIFIED 後申請屋主角色 | `IdentityNFT` tokenId=2 | 後續 phase |
| TENANT | KYC VERIFIED 後申請租客角色 | `IdentityNFT` tokenId=3 | 後續 phase |
| AGENT | KYC VERIFIED 後申請仲介角色 | `IdentityNFT` tokenId=4 | 後續 phase |

### 5.2 平台自動化角色

平台不是「人工 admin 驅動」，而是由 Go service 使用 `OPERATOR_ROLE` 執行受控鏈上操作。

目前最重要的原則：

- 使用者私鑰永遠只在 MetaMask
- 平台不代替使用者送出 MetaMask 交易
- OPERATOR key 只做受控的自動化鏈上操作

### 5.3 OPERATOR key 目前用途

目前已確立的用途：

- `IdentityNFT.mint()`
- `IdentityNFT.rebind()`
- `IdentityNFT.revoke()`

兼容舊骨架的用途仍存在，但不應作為未來架構的主語意。

## 七、資安設計

### 7.1 身份驗證層

| 機制 | 說明 |
|------|------|
| SIWE | 登入與綁錢包都需 SIWE 驗證 wallet ownership |
| Nonce 防重放 | 每次 SIWE 皆使用一次性 nonce |
| Session Cookie | `httpOnly`、`sameSite=Lax`、server-side TTL |
| OTP 驗證 | onboarding 前半段以 Email / SMS OTP 驗證 |

### 7.2 鏈上同步安全

| 風險 | 防護 |
|------|------|
| 前端謊報交易結果 | 以鏈上 event + indexer 為準 |
| 重複事件處理 | `processed_events` 去重 |
| re-org | safe block + checkpoint |
| RPC 故障 | fallback RPC |

### 7.3 私鑰管理原則

- `.env` 管理私鑰與敏感設定
- 嚴禁 hardcode 私鑰或 RPC URL
- OPERATOR key 只做平台自動化，不動用使用者資產
- 未來可升級到 HSM / 多簽，但不是目前第一優先

## 十二、Auth + eKYC 架構（實作現況）

### 12.1 登入機制

正式登入條件：

- `person_hash`
- wallet SIWE
- password

登入成功後由後端建立 `wallet_session`，以 session cookie 維持登入。

### 12.2 Auth 端點

目前正式存在的核心端點：

- `POST /api/auth/login`
- `POST /api/auth/password/set`
- `GET /api/auth/me`
- `POST /api/auth/logout`
- `POST /api/auth/wallet/change`
- `POST /api/auth/wallet/siwe/message`
- `POST /api/auth/wallet/siwe/verify`

### 12.3 Session Cookie

- `httpOnly = true`
- `sameSite = Lax`
- 本機開發 `secure = false`
- 正式環境應改 `secure = true`
- server-side TTL 預設 86400 秒

### 12.4 自建 eKYC 流程（KYC-first Onboarding）

> 目前專案正式採用此流程，這一段是 live flow。

1. Email OTP
2. Phone OTP
3. 上傳身分證正反面
4. OCR 擷取資料
5. 使用者確認資料
6. 上傳第二證件
7. 上傳自拍做人臉比對
8. 綁定唯一錢包 + SIWE
9. 建立 `users` 與 `kyc_submissions`
10. Go OPERATOR key 直接 mint `IdentityNFT`
11. indexer 將 `users.kyc_status` 同步為 `VERIFIED`

### 12.5 OTP 設計

- OTP 存於 `otp_codes`
- target 60 秒限流
- 預設有效期 300 秒
- SMTP / Mitake 未配置時，開發模式以 log-only fallback

### 12.6 身份 Hash

| Hash | 計算方式 | 用途 |
|------|---------|------|
| `person_hash` | `SHA-256(id_number)` | 一人一帳號 |
| `identity_hash` | `SHA-256(person_hash + lower(wallet))` | IdentityNFT reference |

### 12.7 IdentityNFT Token IDs

| tokenId | 名稱 | 狀態 |
|---------|------|------|
| 1 | NATURAL_PERSON | 目前已使用 |
| 2 | OWNER | 後續 phase |
| 3 | TENANT | 後續 phase |
| 4 | AGENT | 後續 phase |
