## 一、核心架構

### 1.1 架構名稱與定位

本平台採用的架構模式在業界有正式名稱，是三個模式的結合：

| 模式 | 傳統軟體領域 | 本平台的對應 |
|------|------------|------------|
| **Event Sourcing** | 以事件日誌為唯一狀態來源 | 鏈上合約 emit events 為 Source of Truth |
| **CQRS** | Command / Query 責任分離 | 寫入走 MetaMask → 鏈；讀取走 Go API → DB |
| **Indexer Pattern** | 事件驅動的讀模型同步 | Go Indexer 訂閱鏈上事件，同步至 PostgreSQL |

Web3 領域通常稱整體架構為 **"On-chain Event Sourcing + CQRS"**，或口語上叫 **"Subgraph Pattern"**（The Graph 協議使用的術語）。

> 口訣：**合約是帳本（Write）、DB 是快取（Read）、Indexer 是搬運工**。

```

└────────────────────┬─────────────────┬────────────────────┘

              ┌──────▼──────┐    ┌──────▼──────┐

              └──────────────┬─────────────────┘

              │    （索引層、查詢快取、文件存放）  │
              └────────────────────────────────┘
```

### 1.3 各層職責

| 層 | 職責 | 不做什麼 |
|----|------|---------|
| **區塊鏈** | 狀態最終裁決者，業務規則執行者 | — |
| **前端（用戶）** | MetaMask 簽名送 tx、UI 展示、Preflight 查詢 | 不直接讀 DB |
| **Go-service** | REST API、Indexer、Hash 計算、Preflight、通知、**平台自動化操作（OPERATOR key）** | 不代替用戶簽名 |
| **AI Agent**（Phase 2） | 物件文件審核、產權真實性判斷，輸出審核結果給 Go OPERATOR 執行 | 不持有用戶資料 |
| **PostgreSQL** | 索引、快取、查詢優化、文件 URL | 不是 Source of Truth |

> **人工介入原則：極小化**。平台後台只處理申訴 / 客服爭議，不介入任何正常業務流程。

### 2.2 啟動流程

`main.go` 只做一件事：呼叫 `bootstrap`。所有 wire 邏輯集中在 `bootstrap/wiring.go`。

```
cmd/server/main.go
  → bootstrap.Run(ctx)

bootstrap/wiring.go（依序執行）：
  1. platform/config    載入 .env
  2. platform/db        連線 PostgreSQL
  3. platform/blockchain 初始化 ethclient（+ failover）
  4. platform/notifier  啟動 WebSocket Hub（goroutine）
  5. db/repository      初始化所有 repo（注入 db 連線）
  6. platform/blockchain/contracts 初始化合約呼叫層（注入 ethclient + OPERATOR key）
  7. modules/xxx        初始化所有 service（注入 repo + contracts）
  8. modules/xxx        初始化所有 handler（注入 service）
  9. platform/indexer   啟動 Indexer，註冊各模組 event_handler 為 worker（goroutine）
 10. bootstrap/router   註冊 Gin 路由，注入 handler
 11.                    啟動 Gin HTTP server（:8080）

優雅關閉（signal.NotifyContext）：
  SIGTERM / SIGINT → ctx cancel → indexer goroutine 停止 → server graceful shutdown
```

### 3.2 `platform/indexer/` + `modules/xxx/event_handler.go` — 鏈上事件索引器

#### 設計原則

```
輪詢模式（HTTP RPC）：每 12 秒輪詢最新 block
↓
扣除確認深度（3 blocks）→ 處理 safe block
↓
逐 block 呼叫 ContractWorker.ProcessBlock()
↓
冪等寫入 DB（ON CONFLICT DO NOTHING）
↓
更新 checkpoint
```

**職責切分：**
- `platform/indexer/`：引擎框架（輪詢、確認深度、冪等、goroutine 管理）
- `modules/xxx/event_handler.go`：各模組自己的事件邏輯（解析 log → 更新 DB）

每個模組的 `event_handler.go` 實作 `ContractWorker` 介面，向 Indexer 引擎註冊，各自獨立 goroutine，互不阻塞。事件業務邏輯與業務 service 放在同一模組，不會分裂。

#### platform/indexer/indexer.go（主協調器）

```go
type ContractWorker interface {
    ContractName() string
    Address() common.Address
    ProcessBlock(ctx context.Context, eth *ethclient.Client, blockNumber uint64) error
}

type Indexer struct {
    client     *blockchain.Client
    checkpoint *CheckpointStore
    workers    []ContractWorker
    cfg        *config.BlockchainConfig
}
```

#### platform/indexer/checkpoint.go（進度記錄）

```go
// DB table: indexer_checkpoints
// columns: contract_name, last_processed_block, updated_at

func (s *CheckpointStore) GetLastBlock(ctx, contractName) (uint64, error)
func (s *CheckpointStore) SaveBlock(ctx, contractName, blockNumber) error
func (s *CheckpointStore) MarkProcessed(ctx, txHash, logIndex, contractName, eventName, blockNumber) (isNew bool, err error)
```

#### modules/property/event_handler.go（以 PropertyRegistry 為例）

```go
func (w *PropertyWorker) ProcessBlock(ctx context.Context, eth *ethclient.Client, blockNumber uint64) error {
    logs := eth.FilterLogs(ctx, FilterQuery{
        Addresses: []common.Address{w.Address()},
        FromBlock:  big.NewInt(int64(blockNumber)),
        ToBlock:    big.NewInt(int64(blockNumber)),
    })

    for _, log := range logs {
        // 冪等處理：同一 tx_hash + log_index 不重複寫入
        isNew, _ := checkpoint.MarkProcessed(log.TxHash, log.Index, ...)
        if !isNew {
            continue
        }

        switch log.Topics[0] {
        case PropertyRegisteredTopic:
            // 更新 properties 表：status = REGISTERED
        case PropertyVerifiedTopic:
            // 更新 properties 表：verification_status = VERIFIED
        case DisclosureUpdatedTopic:
            // 更新 properties 表：disclosure_hash
        }
    }
    return nil
}
```

## 四、完整資料流範例

### 4.1 KYC 驗證流程（TWID → SBT，全自動）

```
1. 前端 POST /api/kyc/twid/start
   → Go 向 TWID API 取得驗證 URL（帶 state=wallet_address）
   → 回傳 URL 給前端

2. 前端跳轉 TWID，使用者完成身份驗證（身分證 + 手機 + 活體）

3. TWID POST /api/kyc/twid/callback（打到 Go）
   → Go 驗證 TWID HMAC 簽章（防偽造）
   → 取得 kyc_reference_id，驗證一人一帳號
   → Go OPERATOR key 自動呼叫 IdentityNFT.mint()
   → 不需要人工介入，TWID 已是可信第三方

4. Indexer 收到 IdentityMinted event
   → DB: users.kyc_status = VERIFIED, identity_nft_token_id = tokenId
   → Notifier 推送 "kyc_verified" 給使用者前端
```

### 4.2 物件登記流程（MVP：規則自動審核；Phase 2：AI Agent 審核）

```
1. 前端上傳產權文件 POST /api/properties/upload-deed
   → Go 計算 deed_hash（blockchain.DeedHash）
   → Preflight：deed_hash 不得重複；呼叫者 kyc_status == VERIFIED
   → 回傳 { deedHash, propertyId }

2. 前端用 MetaMask 呼叫 PropertyRegistry.registerProperty(propertyId, deedHash)

3. Indexer 收到 PropertyRegistered event
   → DB: properties 新增記錄（status = REGISTERED）

4. 自動審核流程：
   MVP：Go 檢查基本規則（格式、hash 唯一性、KYC 狀態）通過後自動 verifyOwner + verifyProperty
   Phase 2：AI Agent 分析產權文件（OCR + 模型判斷）→ 輸出信心分數 → Go OPERATOR key 執行

5. Go OPERATOR key 自動呼叫：
   PropertyRegistry.verifyOwner(propertyId, ownerWallet)
   PropertyRegistry.verifyProperty(propertyId)

6. Indexer 收到 OwnerVerified + PropertyVerified events
   → DB: owner_verified = true, verification_status = VERIFIED
   → Notifier 推送 "property_verified" 給屋主
```

### 4.3 前端送 tx 前的 Preflight 驗證

```
// 前端送 MetaMask tx 前，先問後端

GET /api/preflight/register-property?deedHash=0x...
→ Go 查 DB：deed_hash 是否重複？
→ Go 查 DB：呼叫者 kyc_status 是否 VERIFIED？
→ 回傳 { ok: true } 或 { ok: false, reason: "..." }
→ 前端根據結果決定是否送 tx（避免浪費 gas）
```

## 五、業務規則模型

### 5.1 角色與權限矩陣

> 角色是**用戶側**的概念，不存在「平台 Admin 角色」。平台自動化操作由 Go OPERATOR key 執行，不是一種用戶角色。

| 角色 | 取得方式 | 鏈上憑證 | 可執行操作 |
|------|---------|---------|----------|
| **未登入** | 無需 | 無 | 瀏覽公開刊登、看市場看板 |
| **USER**（已登入） | SIWE 登入 | 無 | 提交 KYC 申請 |
| **KYC VERIFIED** | TWID 驗證 → Go OPERATOR 自動 mint | `IdentityNFT` SBT | 登記物件、提交意向 |
| **OWNER** | KYC VERIFIED + 產權文件審核通過 | `IdentityNFT` + `PropertyRegistry` | 揭露現況、授權仲介、刊登物件、新增共同持有人 |
| **AGENT** | KYC VERIFIED + 執照欄位填寫 | `IdentityNFT` + agent fields | 接受授權後刊登他人物件 |

> **Go OPERATOR key** 是平台的自動化執行機制，不是用戶角色。它呼叫的是 `OPERATOR_ROLE` 合約函式，觸發條件都是業務規則自動達成，不需要任何人工確認。

### 5.2 強屋主驗證模型（Owner Credential）

> **KYC 是「你是誰」；強屋主驗證是「你對這個物件有沒有權」**

```
強屋主驗證 = KYC VERIFIED + 產權文件審核通過（自動化）

驗證流程：
屋主上傳產權文件（PDF / 影本）
  → Go 計算 deed_hash，Preflight 驗重複性與 KYC 狀態
  → PropertyRegistry.registerProperty(propertyId, deedHash) ← 用戶 MetaMask

  → 【MVP】Go 執行基本規則審核：
      deed_hash 唯一 + KYC VERIFIED + 文件格式合法
      → 條件通過，Go OPERATOR key 自動呼叫 verifyOwner + verifyProperty

  → 【Phase 2】AI Agent 審核：
      AI 分析產權文件（OCR + 不動產模型）→ 輸出信心分數
      → 信心分數達閾值，Go OPERATOR key 自動呼叫 verifyOwner + verifyProperty
      → 信心分數不足，退回用戶補件（自動通知）

  → Indexer 收到 OwnerVerified + PropertyVerified events
  → DB: owner_verified = true, verification_status = VERIFIED

關鍵設計：
- deed_hash 在 DB 及合約中均設 UNIQUE，一個產權文件只能登記一次
- verifyOwner 在合約上對 owner wallet 打標，鏈上可查
- 全流程自動化，無人工介入，鏈上記錄審核結果可公開追溯
- 驗證後才可授權仲介或上架刊登
```

### 5.3 共同持有模型（Co-ownership）

```
DB: property_owners（關聯表）
  property_id         BIGINT FK → properties
  owner_wallet_address VARCHAR
  share_bps           SMALLINT    （1000 = 10%，加總必須 = 10000）
  is_primary_operator BOOLEAN     （同時只有一人為 true）
  owner_verified      BOOLEAN     （Go OPERATOR key 執行 verifyOwner 後設為 true）

合約: PropertyRegistry
  mapping(bytes32 => mapping(address => OwnerInfo)) owners
  mapping(bytes32 => address) primaryOperator

規則：
- 第一個 registerProperty 的人自動成為 primary operator
- Primary operator 可呼叫 addOwner 新增共同持有人
- addOwner 後由 Go OPERATOR key 自動執行 verifyOwner（審核邏輯同上）
- setPrimaryOperator 可轉移主操作人（由現任 primary 操作）
- 揭露義務、授權仲介只有 primary operator 可執行
- 未來擴充：多簽機制（需達 N/M 同意才可授權）
```

### 5.4 物件揭露義務模型（Disclosure Obligation）

> 這是平台最有差異化的功能：強制屋主在上架前聲明物件現況

```
揭露欄位（disclosure schema）：
  is_haunted          BOOLEAN   凶宅
  has_leak            BOOLEAN   漏水
  has_dispute         BOOLEAN   糾紛
  is_occupied         BOOLEAN   現居住狀態
  other_notes         TEXT      其他重大瑕疵

上鏈方式：
  disclosureHash = SHA-256(is_haunted | has_leak | has_dispute | is_occupied | notes)
  PropertyRegistry.updateDisclosureHash(propertyId, disclosureHash) ← MetaMask

DB 欄位：
  disclosure_is_haunted / has_leak / has_dispute / is_occupied  個別欄位（方便查詢篩選）
  disclosure_hash         VARCHAR  對應鏈上 hash（Indexer 同步）
  disclosure_completed    BOOLEAN  四個必填欄位都填過才 true

業務規則：
  - 刊登前必須 disclosure_completed = TRUE
  - Preflight API 強制驗證
  - 事後更新揭露：需重新呼叫 updateDisclosureHash → Indexer 同步 → 通知潛在租客
  - 虛假揭露：平台可 slashStake + forceRevoke 授權，作為懲罰接口
```

## 六、Case 狀態機（最終定版）

### 6.1 租屋（RENT）狀態流

```
OPEN ──→ MATCHED ──→ SIGN ──→ CLOSED
  └──────────────────────────→ CANCELLED
```

| 狀態 | 觸發條件 | 合約函式（Go OPERATOR key 自動執行） | DB 欄位 |
|------|---------|--------------------------------------|---------|
| `OPEN` | 雙方意向確認 | `CaseTracker.openCase()` | `opened_at`, `open_tx_hash` |
| `MATCHED` | 屋主收訂金確認 | `CaseTracker.advanceCase(MATCHED)` | `matched_at`, `matched_tx_hash` |
| `SIGN` | 進入簽約流程 | `CaseTracker.advanceCase(SIGN)` | `sign_at`, `sign_tx_hash` |
| `CLOSED` | 結案條件達成 | `CaseTracker.advanceCase(CLOSED)` | `closed_at`, `closed_tx_hash` |
| `CANCELLED` | 任一方取消或申訴裁定 | `CaseTracker.cancelCase()` | `cancelled_at`, `cancel_tx_hash` |

### 6.2 買賣（SALE）狀態流

```
OPEN ──→ MATCHED ──→ SIGN ──→ BANKING ──→ CLOSED
  └──────────────────────────────────────→ CANCELLED
```

| 狀態 | 說明 |
|------|------|
| `OPEN` | 刊登公開，接受看屋預約 |
| `MATCHED` | 媒合成功，雙方意向確認 |
| `SIGN` | 簽約中（買賣合約） |
| `BANKING` | 貸款申辦中（買賣獨有） |
| `CLOSED` | 完成交割 |
| `CANCELLED` | 任一階段取消（附原因） |

### 6.3 OPEN 狀態熱度指標

```sql
-- listings 表
reservation_count   INT   -- 已提交看屋預約數
view_count          INT   -- 已查看詳情次數
```

前端在 OPEN 狀態卡片上顯示：`👁 12 次查看 | 📅 3 個預約`

## 七、完整資料流範例（Case）

### Case 開立流程

```
1. Listing 進入媒合候選
   → 前端：承租人 / 買家提交意向（POST /api/cases/intent）
   → 後端：DB 記錄意向，推送通知給屋主 / 刊登人

2. 雙方意向確認 → 業務條件達成
   → Go OPERATOR key 自動呼叫 CaseTracker.openCase(...)
   → 全程無需人工確認，條件即觸發

3. Indexer 收到 CaseOpened event
   → DB: cases 新增記錄，status = OPEN
   → Notifier 推送 "case_opened" 給雙方

4. 每次狀態推進（MATCHED → SIGN → CLOSED）
   → 對應業務條件達成（如屋主確認收訂）
   → Go OPERATOR key 自動呼叫 CaseTracker.advanceCase(caseId, newStatus)
   → Indexer 同步 → DB 更新對應 *_at + *_tx_hash
   → Notifier 推送狀態更新給雙方

5. 結案後
   → Go OPERATOR key 自動呼叫 ListingStakeVault.releaseStake(listingId)
   → Indexer 同步 → DB: listings.stake_tx_hash 更新
```

## 八、資安設計

### 8.1 身份驗證層

| 機制 | 說明 |
|------|------|
| **SIWE** | 所有寫入 API 需要 wallet session（已實作） |
| **Nonce 防重放** | 每個 SIWE 請求使用一次性 nonce（已實作） |
| **Session 過期** | Session token 有時效，強制重新簽名 |
| **TWID HMAC** | callback 必須驗 TWID 的 HMAC 簽章，防偽造 |

### 8.2 區塊鏈互動資安

| 威脅 | 防護措施 |
|------|---------|
| **前端謊報 tx** | Indexer 直接從鏈上讀取，不信任前端回傳的狀態 |
| **Re-org（鏈重組）** | `processed_events` 表記錄 block_num，偵測並回滾 |
| **重複事件處理** | `PRIMARY KEY (tx_hash, log_index)` 保證冪等 |
| **RPC 節點失效** | failover 至備用 RPC，exponential backoff |
| **ABI 版本不符** | ABI JSON 從 hardhat artifacts 複製，版本鎖定 |

### 8.3 API 資安

| 威脅 | 防護措施 |
|------|---------|
| **DDoS / 暴力攻擊** | 雙層 Rate Limit：IP + Wallet Address |
| **Wallet 級別限流** | 每個 wallet 每分鐘最多 N 次寫入請求 |
| **SQL Injection** | Parameterized query（lib/pq 已防護） |
| **敏感資料外洩** | DB 不存身分證號、姓名；Log 不輸出 PII |
| **CORS** | 嚴格限制 `AllowOrigins` 為已知前端域名 |

### 8.4 私鑰與角色管理原則

合約中共有四個明確語意的角色，**每個角色讀名字即知職責**：

| 角色 | 持有者 | 職責 | 注意 |
|------|--------|------|------|
| `OPERATOR_ROLE` | Go-service 環境變數 | 平台自動化業務操作（mint, verify, openCase…） | 不接觸用戶資金 |
| `TREASURY_ROLE` | 財務獨立錢包 | 提領累積服務費與沒收押金 | 不操作業務 |
| `EMERGENCY_ROLE` | 冷錢包（部署者） | 緊急提領全部資金（合約升級 / 意外） | 正常情況不使用 |
| `DEFAULT_ADMIN_ROLE` | 部署者（OZ 內部） | OpenZeppelin 角色管理機制，不綁任何業務函式 | 僅用於 grantRole / revokeRole |

```
用戶私鑰：     MetaMask 本地管理（平台永遠碰不到）
OPERATOR key： 環境變數注入（.env），不 hardcode
               → 只有 OPERATOR_ROLE 權限，無法動用用戶資金
               → 未來可升級為 HSM（硬體安全模組）或多簽
TREASURY key： 獨立財務錢包，只能提領已累積的平台收入
EMERGENCY key：冷錢包，正常情況永不使用，僅限合約升級緊急情境
```

**OPERATOR key 只做平台業務，不代替用戶**：
```
✓ mint KYC SBT（TWID 已驗證）
✓ verifyOwner / verifyProperty（AI Agent 已審核）
✓ openCase / advanceCase（業務條件達成）
✓ releaseStake / slashStake（結案 / 違規條件達成）
✗ 代替用戶 registerProperty（不允許）
✗ 代替用戶 createListing（不允許）
✗ 代替用戶 payServiceFee（不允許）
✗ 動用任何用戶資金（不允許）
```

## 九、Go 在這個架構中的完整價值

| Go 模組 | 對應 Web3 能力 | 為什麼 Go 最適合 |
|---------|--------------|----------------|

| `twid/` | OAuth callback + HMAC 驗證 | 標準 `crypto/hmac` + 快速 HTTP |
| Preflight API | Tx 前置驗證閘道 | 低延遲 API，快速回應前端 |
| Rate Limiter | Wallet 級別限流 | `sync.Map` + `time.Ticker` 高效實作 |
