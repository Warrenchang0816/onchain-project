## 1. 目前鏈上主線與邊界

本專案的鏈上設計已轉向房屋平台主線，但截至 2026-04-22，正式接線狀態要分開看：

- 已正式接線：
  - `IdentityNFT` 自然人 KYC 憑證
  - Go indexer 對 `IdentityMinted` 的同步
  - `users.kyc_status` 與 `identity_nft_token_id` 的回寫
- 已有合約骨架與部署腳本，但尚未全部閉環：
  - `PropertyRegistry`
  - `AgencyRegistry`
  - `CaseTracker`
  - `ListingStakeVault`
- 仍屬舊相容或保留內容：
  - `TaskRewardVault` 舊任務系統相關變數與流程

主幹 Gate 對應：

- `Gate 1`：角色憑證閉環
- `Gate 2`：Property On-chain
- `Gate 3`：Agency On-chain
- `Gate 4`：Case On-chain
- `Gate 5`：Stake 正式上鏈

## 2. 鏈上互動角色

目前專案有兩種主要鏈上角色：

| 角色 | 誰在操作 | 主要責任 |
|------|----------|----------|
| 使用者錢包 | 前端 MetaMask | SIWE、未來的 property / agency / stake 寫入交易 |
| 平台 Operator | Go 後端 | KYC mint、驗證、案件推進、事件同步治理 |

補充原則：

- 私鑰只應存在後端或部署環境，不得放進前端。
- 前端不應把「交易已送出」當成最終成功；正式狀態仍以 event / indexer / read model 為準。

## 4. `.env` 對齊方式

### 4.1 `task-reward-vault/.env`

主要欄位：

- `SEPOLIA_RPC_URL`
- `SEPOLIA_PRIVATE_KEY`
- `TREASURY_ADDRESS`
- `OPERATOR_ADDRESS`
- `USDC_ADDRESS`
- `USDT_ADDRESS`

用途：

- 編譯與部署合約
- 決定 deployer / operator / treasury
- 決定 `ListingStakeVault` 的穩定幣白名單

### 4.2 `go-service/.env`

目前鏈上相關主欄位：

- `APP_CHAIN_ID`
- `APP_RPC_URL`
- `APP_PLATFORM_OPERATOR_PRIVATE_KEY`
- `ETH_RPC_URL_FALLBACK`
- `ETH_CONFIRMATION_BLOCKS`
- `IDENTITY_NFT_ADDRESS`
- `IDENTITY_NFT_START_BLOCK`
- `PROPERTY_REGISTRY_ADDRESS`
- `AGENCY_REGISTRY_ADDRESS`
- `LISTING_STAKE_VAULT_ADDRESS`
- `CASE_TRACKER_ADDRESS`
- `USDC_ADDRESS`
- `USDT_ADDRESS`

補充說明：

- `APP_REWARD_VAULT_ADDRESS` 等舊任務追蹤欄位仍保留相容，但不是目前房屋平台主幹的主要設定。
- 若只驗證目前已接線主線，最重要的是 `IdentityNFT` 與其 start block。

### 4.3 `react-service/.env`

目前前端主線實際用到的重點欄位：

- `VITE_API_GO_SERVICE_URL`
- `VITE_CHAIN_ID`
- `VITE_CHAIN_NAME`
- `VITE_RPC_URL`
- `VITE_EXPLORER_URL`

補充說明：

- `react-service/.env.example` 仍保留部分舊任務系統相容欄位；這些不是目前房屋平台主幹的正式前端準據。
- Gate 2 之後若有更多前端直連合約頁面，再補對應合約地址環境變數即可。

## 5. 合約部署流程

### 5.1 安裝依賴

```powershell
cd D:\Git\onchain-project\task-reward-vault
Copy-Item .env.example .env
npm install
```

### 5.2 編譯

```powershell
npm run compile
```

### 5.3 部署房屋平台合約組

```powershell

```

對應腳本：

- `task-reward-vault/scripts/deployHousePlatform.ts`

- `IDENTITY_NFT_ADDRESS`

- `go-service/.env`

- `react-service/.env`

## 6. 目前正式接線的鏈上工作流

### 6.1 自然人 KYC / IdentityNFT

目前正式可描述為 live flow 的鏈上主線是：

1. 使用者完成 eKYC
2. Go Operator 送出 `IdentityNFT` mint
3. indexer 掃描 `IdentityMinted`
4. DB 更新 `users.kyc_status = VERIFIED`
5. 身份中心 / 會員資料頁讀取 read model 顯示狀態

### 6.2 尚未可描述為已完成的部分

以下合約能力雖已存在設計或腳本，但目前不能寫成已正式閉環：

- OWNER / TENANT / AGENT 的 `CredentialMinted` 閉環
- Property 驗證與 `PropertyRegistry` read model
- Agency 授權狀態同步
- CaseTracker 正式案件流
- ListingStakeVault 正式 stake / slash 治理

## 7. 安全原則

- 嚴禁把任何私鑰、RPC key、SMTP / AWS 憑證寫死在程式碼內。
- `SEPOLIA_PRIVATE_KEY` 與 `APP_PLATFORM_OPERATOR_PRIVATE_KEY` 都屬高敏感資訊。
- 前端 `.env` 視同公開，不能放任何 secret。
- 所有鏈上寫入後的正式狀態，要靠 event / indexer / DB read model 驗證，不可只信任前端回傳。

## 9. 常見問題

### 為什麼這份指南不再用 `createAndFundTask` / `claimReward`

因為那些是舊任務追蹤系統的語境，不是目前房屋平台主幹。房屋平台現在的主幹是身份、物件、授權、案件與 stake 治理。

### 為什麼合約都列出來了，但文件又說還沒正式閉環

因為「有合約骨架 / 可以部署」不等於「後端 indexer、DB read model、前端頁面與驗收流程都已完成」。目前主幹仍需依 Gate 順序收斂。

### 如果只想跑目前平台主線，需要先部署全部 5 個合約嗎

不用。若只是跑目前已接線主線，`IdentityNFT` 與 KYC 流程才是最優先的鏈上部分，其餘合約屬後續 Gate。
